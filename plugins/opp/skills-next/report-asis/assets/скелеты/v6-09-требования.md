# Скелет раздела: Бизнес-требования (тело, раздел 09)

Инструкция агенту-редактору: собрать нарратив о бизнес-требованиях ИЗ памяти. Требование —
проверяемое условие целевого решения с критерием приёмки и бизнес-результатом; требования служат
основой технического задания. Группировка и порядок — как у проблем и решений.

## Что читать из памяти
- **TB** (требования): `формулировка`, `критерий приёмки`, `бизнес-результат`, `устраняемый ОБЭ`,
  `покрывает проблемы (полностью)`, `покрывает проблемы (частично)`, `статус требования`.
- **SOL**: `закрываемое требование` — для трассировки «требование → решение».
- **PRB**: покрываемые проблемы — для трассировки «требование → проблема».
- Держи сквозную трассировку требование → решение → проблема; каждое требование — в одной группе
  (по основной проблеме), смежные проблемы других групп — в поле трассировки.

## Структура и порядок изложения
1. Вводный абзац (без подзаголовка): что такое требование в этом отчёте (критерий приёмки +
   бизнес-результат), счёт требований, принцип группировки и сквозной трассировки, общий статус.
2. По каждой группе — `### Требования группы: <название группы>`:
   - 1 абзац: к какому замыслу решения группы относятся требования.
   - **таблица**: `| Код | Требование | Критерий приёмки | Бизнес-результат | Трассировка |`; в
     колонке трассировки — покрываемые проблемы и реализующие решения (читаемыми шифрами).
3. Сводная таблица по всем требованиям в конце (если так принято в проекте).

## Альтитуда
Тело — требования с приёмкой и трассировкой. Не разворачивать проектную реализацию требования: это
следующий такт. Формулировка и критерий — в одну ячейку каждый.

## Правила формулировок
- Экспертный русский требований: формулировка — «Обеспечить/Автоматизировать/Исключить <объект>»;
  критерий приёмки — проверяемый («по любой позиции доступны …», «ручная сверка отсутствует»).
- Коды требований/проблем/решений — читаемые шифры проекта (например «БТ01», «П07», «Р06»), НЕ
  внутренние ключи памяти; `Ур.N` в текст не выносить.
- Обезличенные обороты: «требования охватывают все три типа решений, а не только автоматизацию»,
  «часть закрепляет правила учёта и регламенты, часть касается переноса процессов в систему».

## Честность
Только требования из памяти со своим статусом. Критерий приёмки не выдумывать — если поле пустое,
пометить «критерий приёмки уточняется при проработке ТЗ», а не сочинять. Трассировку давать по
фактическим ссылкам TB/SOL, без домысленных связей.

## Целевой объём
Пропорционально числу групп: ~0.4–0.6 страницы на группу (подводка + таблица).
