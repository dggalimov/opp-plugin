# Навык: Отчёт об обследовании AS-IS

> Навык ПРОДУКТА. Отчёт — детерминированная проекция памяти. Проза в разделах ЕСТЬ, но каждый её
> факт взят из памяти (доказуемость), а структура задана скелетом раздела (повторяемость).
> Нарративы пересобираются каждым прогоном — это производная памяти, не рукопись. Готовый
> документ руками не правят: правда живёт в памяти и в скелетах, документ пересобирается.

## Шаг 0. Предусловия (без них отчёт не выдаётся)

1. `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR (замкнутость, гейт ОБЭ, трассировка).
2. `"${CLAUDE_PLUGIN_ROOT}/opp" prove --into "<проект>"` — 0 дефектов доказательств (величины
   воспроизводимы или честно пусты).
3. `"${CLAUDE_PLUGIN_ROOT}/opp" requirements --into "<проект>"` — 0 непокрытых проблем и требований
   без проблемы (или расхождения объяснены в обзоре).
4. **Картина каркаса синтезирована:** `"${CLAUDE_PLUGIN_ROOT}/opp" picture --into "<проект>"` —
   уровни не ниже «готов к синтезу», критичных открытых вопросов нет (обход — только явным
   допущением владельца). Отчёт AS-IS — проекция синтезированного каркаса, а не полуразведки.

Дефекты есть → сначала чинится память (навыком синтеза/каналов), потом отчёт.

## Шаг 1. Нарративы разделов (пишутся ПЕРЕД рендером)

Для каждого из 13 разделов отчёта (9 тела + 4 приложений; состав и файлы — `documents/asis/config.yaml`):

1. **Прочти скелет раздела** из `${CLAUDE_PLUGIN_ROOT}/skills/report-asis/assets/скелеты/<имя-файла>.md`
   (имя совпадает с полем `файл:` секции). Если в проекте есть замещающий скелет-шаблон
   `<ws>/шаблоны документов/скелеты-asis/<имя-файла>.md` — он приоритетнее продуктового (механизм
   «шаблоны документов»: владелец правит СТРУКТУРУ раздела здесь, не в данных).
2. **Прочти указанные скелетом таблицы памяти** (коды и поля перечислены в скелете) из
   `<ws>/Проектная память/…` через `"${CLAUDE_PLUGIN_ROOT}/opp"`-инструменты чтения памяти.
3. **Напиши нарратив** `<ws>/нарратив/asis/<имя-файла>.md` строго по скелету: его структура (`###`
   подзаголовки, где таблицы-выжимки, где связный текст), альтитуда (тело — обобщение, детали — в
   приложениях), правила формулировок и объём. Каждый факт нарратива — из прочитанной памяти;
   ничего от себя. Дыры памяти оформляй честной фразой скелета («не обследовано, вынесено в
   открытые вопросы»), а не домыслом.

Пересборка полная: старые `нарратив/asis/*.md` перезаписываются этим прогоном (они — производная
памяти). Правки владельца по ФАКТАМ идут в память (каналы/синтез) → пересборка; по СТРУКТУРЕ —
в скелет-шаблон проекта → пересборка.

## Шаг 2. Рендер

```bash
"${CLAUDE_PLUGIN_ROOT}/opp" render asis --into "<проект>" --docx
```

Выход: md-канон в техслое `<ws>/Проектная память/Документы/asis.md` + клиентский
`<ws>/Документы/Отчёт об обследовании AS-IS.docx` (с темой оформления).

## Шаг 3. Самосверка полноты (перед выдачей)

- **Каждый раздел непуст.** В md-каноне нет строки `_Раздел не подготовлен._` — значит все 13
  нарративов на месте. Пустой раздел = не написанный на шаге 1 нарратив, дописать.
- **Коды памяти не протекли.** `grep -E 'SRC-|PRB-|MTG-|EFF-|TB-|SOL-|DEC-|AM-|SYS-|PRC-|OQ-|HYP-|VP-|STK-|BR-|RM-|MAT-' <ws>/нарратив/asis/*.md`
  должен дать 0 совпадений — в клиентском тексте коды заменяются читаемыми шифрами/именами.
- Счётчики (проблемы/решения/требования, встречи/материалы) в нарративах сходятся со счётом строк
  памяти. Величины ущерба: пустое — честное «ожидает данных заказчика», НЕ дефект, не заполнять.

## Шаг 4. Выдача и правки

Документ уходит заказчику/владельцу на просмотр. Замечания по СОСТАВУ разделов/виду — правка
`documents/asis/config.yaml` (+ golden пересчитать). По СТРУКТУРЕ раздела — правка скелета-шаблона
проекта (шаг 1) и пересборка. По ФАКТАМ — через память (правки протоколов каналов
`"${CLAUDE_PLUGIN_ROOT}/opp" split` или навыки синтеза/каналов), затем пересборка нарративов и
отчёта. Прямое редактирование asis.md/docx запрещено — потеряется при следующей сборке.

## Границы

НЕ меняет память; проза разделов пишется ТОЛЬКО из фактов памяти по скелету (не свободное
сочинение); НЕ выдаёт отчёт при красных предусловиях или несинтезированной картине; PDF и
брендированные шаблоны — вне финала.
