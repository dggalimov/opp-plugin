# Навык: Решения и подбор продуктов

> Навык ПРОДУКТА. Решение — не хотелка и не каталог: каждое отвечает конкретному требованию,
> каждый названный продукт опирается на знание с источником или честно помечен белым пятном.

## Несущие правила

1. **Решение отвечает требованию.** Каждая строка SOL ссылается на TB (одно решение может
   закрывать несколько требований); решений «из воздуха» нет. Каждое TB закрыто хотя бы
   одним SOL — `"${CLAUDE_PLUGIN_ROOT}/opp" solutions` проверит.
2. **Тип — по триаде и согласован с уровнем** (C6): методологический дефект не лечится
   «внедрить модуль»; сначала методика/оргмера, технология — где корень технологический.
3. **Продукт — только из знаний с источником.** Профиль конфигурации проекта (1С:ERP,
   БИТ.Финанс, 1С:УХ, 1С:ЗУП) + доступные БЗ продукта. Нет БЗ по функции — пиши решение
   с пометкой «покрытие требует проверки (демо/пилот)» — это белое пятно, не утверждение.
   Не выдумывай возможности продуктов.
4. **Выбор платформы — DEC, не приговор.** Ключевые развилки проекта (например, выбор
   финансового контура: развитие текущей платформы против перехода на целевую) оформляются
   строкой DEC: критерии, факты из памяти за/против (демо-встречи, техдолг, восприятие
   пользователей), взвешивание. Вердикт «требует решения заказчика», если фактов
   недостаточно — не решай за заказчика его стратегию.
5. **Дедуп и слаги** — как везде: «уже_в_памяти» прежде чеканки, коды по словарю
   `${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/словарь-слагов.md`.

## Процедура

1. Вход: `"${CLAUDE_PLUGIN_ROOT}/opp" requirements --into "<проект>"` — реестр TB чист (непокрытых проблем нет).
   Прочитай TB и их проблемы/ущербы (extract даёт предшественников), профиль конфигурации
   и `обзор.md`.
2. Сгруппируй требования по контурам (бюджет, факт УУ, ФОТ, казначейство, инвестиции, НСИ,
   закупки…) — решение обычно закрывает контур, не одиночное требование.
3. `"${CLAUDE_PLUGIN_ROOT}/opp" extract SOL --into "<проект>"` → чеканка решений по правилам 1–3.
4. Развилки платформ → `"${CLAUDE_PLUGIN_ROOT}/opp" extract DEC` → строка(и) DEC по правилу 4.
5. Самопроверка: `"${CLAUDE_PLUGIN_ROOT}/opp" solutions --into "<проект>"` — 0 дефектов (непокрытых TB нет,
   решений без требования нет, типы согласованы); `"${CLAUDE_PLUGIN_ROOT}/opp" lint` — 0 ERROR.
6. Обнови `обзор.md`: решений, покрытие требований, белые пятна знаний, развилки DEC.

## Запись строк — только через рельсы

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<PY
import sys; sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")
import json
from schema.extract import write_rows
problems = write_rows("<КОД>", json.load(open("rows.json", encoding="utf-8")), "<проект>")
for p in problems: print(p)
PY
```

## Границы

НЕ строит TO-BE и дорожную карту (следующие навыки); НЕ считает деньги (КРП вне финала);
НЕ выносит вердикт по платформе за заказчика; НЕ утверждает покрытие без БЗ-источника.
