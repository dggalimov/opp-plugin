# OPP — Руководство пользователя

Плагин для обследования проектов внедрения 1С / ERP: один архитектор ведёт проект вместе с ИИ —
от приёма материалов до отчёта об обследовании и концепции целевого решения.

*Черновик для внутреннего использования. Продукт в развитии — часть шагов помечена «в разработке».*

## Оглавление

Автоматическая (поле TOC в docx).

## 1. Что такое OPP и зачем он

OPP — рабочее пространство и набор навыков, которые помогают архитектору провести обследование
текущего состояния (AS-IS) проекта внедрения 1С/ERP не в одиночку, а в паре с ИИ. Вы задаёте
направление и принимаете решения; ИИ читает материалы и записи встреч, раскладывает факты по
проектной памяти, сводит их в проблемы и требования и собирает документы.

### Что получается на выходе

Пакет документов для заказчика и проектной команды, каждый — детерминированно собранный из
проектной памяти:

- **Протокол аудита предоставленных материалов;**
- **Протоколы встреч** — по каждой встрече, с содержанием под тип встречи (системы / процессы /
  организационная);
- **Отчёт об обследовании AS-IS:** проблемы с доказанным ущербом, бизнес-требования, картина по
  уровням;
- **Концепция TO-BE и дорожная карта:** решения, целевая архитектура, волны и оценка (по мере
  готовности навыков).

### Два принципа, на которых стоит продукт

**Доказуемость.** В готовый документ идёт только подтверждённый факт с указанием источника.
Домысел не выдаётся за факт: он помечается как гипотеза, привязывается к источнику и
перепроверяется. Не подтверждено — либо подтвердить из источника, либо не включать.

**Повторяемость.** Документ — детерминированная проекция зафиксированных фактов: одни и те же
факты дают один и тот же документ. Поэтому документы не правятся руками — правится память,
документ пересобирается.

## 2. Как это устроено (ментальная модель)

> **Если работаете так впервые.** Главный сдвиг: знание проекта живёт не в вашей голове и не в
> файлах Word, а в проектной памяти — наборе таблиц фактов. Вы наполняете память фактами, а
> документы получаете из неё сборкой. Это и даёт доказуемость и повторяемость.

### Проектная память — таблицы фактов

Всё, что узнаётся о проекте, раскладывается по типизированным таблицам: источники, объёмные
параметры, учётные модели, бизнес-правила, процессы, системы, доработки, проблемы, требования и
другие. У каждого факта есть провенанс — ссылка на источник и точное место в нём (лист, раздел,
таймкод записи). Факт без источника недействителен.

### Конвейер: от источника до целевого решения

Факты движутся по стадиям, и порядок не случаен — ранние стадии питают поздние:

**источник → факт (с провенансом) → проблема → доказанный ущерб → требование → решение →
целевая архитектура и дорожная карта**

Обратные связи (например, какая проблема каким требованием закрыта) выводятся автоматически —
их не нужно вести вручную с двух сторон.

### Документы — проекции памяти

Документ собирается из памяти по заданной структуре. Тело документа руками не редактируется:
если в отчёте что-то не так — исправляется факт в памяти, и документ пересобирается. Правки
заказчика возвращаются в память через сверку (команда `split`), а не прямым редактированием.

## 3. Папки проекта: что куда класть

У проекта два слоя папок — пользовательский (приёмные папки, куда вы кладёте свои файлы) и
технический (куда ИИ раскладывает разобранные факты). Разделение важно: приёмные папки — это
архив исходных файлов «как есть», технический слой — это уже разобранное содержимое в виде
узлов-источников.

- **«Входные материалы/»** — сюда кладутся документы заказчика: ТКП, анкеты, регламенты,
  выгрузки, презентации — всё, что не запись встречи. Файлы, присланные вложением в чат, навык
  сам сначала копирует сюда (это архив), а затем принимает командой `ingest`.
- **«Встречи/»** — сюда кладутся записи и материалы встреч (аудио, видео, транскрипты). Тот же
  принцип: вложения в чат сначала копируются сюда, потом обрабатываются.
- **«Источники/»** — технический слой: узлы-источники (md-файлы с разобранным содержимым и
  провенансом), на которые дальше ссылаются факты памяти. Это не пользовательская папка — сюда
  ходить не нужно, ИИ ведёт её сам через `ingest`.

Проще говоря: свои файлы вы кладёте во «Входные материалы» и «Встречи»; «Источники» — это уже
результат разбора, живущий по контракту продукта.

## 4. Быстрый старт

Минимальный путь, чтобы увидеть первый результат на своём проекте.

| № | Шаг | Что сделать |
|---|---|---|
| 1 | **Установить плагин** | Подключить marketplace OPP и включить плагин в Claude Code. |
| 2 | **Развернуть проект** | `./opp deploy "<Клиент> — OPP"` — создаётся рабочая папка: приёмные «Входные материалы/» и «Встречи/», технические «Источники/», «Проектная память/», «Документы/», паспорт и правила проекта. Онбординг `opp:new-project` проведёт по этому шагу и следующим в диалоге. |
| 3 | **Заполнить паспорт** | Внести имя клиента, домен, каналы материалов в «Паспорт проекта» (навык может спросить в диалоге). |
| 4 | **Положить материалы** | Сложить файлы заказчика во «Входные материалы/» (или прислать вложением в чат — навык сам скопирует их туда). |
| 5 | **Принять материалы** | Один раз запустить `./opp ingest` — материалы попадут в реестр источников; повторный прогон безопасен: уже принятые файлы пропускаются. |
| 6 | **Посмотреть, что принято** | `./opp ui` открывает окно проекта в браузере — проверить реестр источников, убедиться, что ничего не потеряно и нет дублей. |

> **Приём материалов идемпотентен.** Повторный запуск `ingest` на тех же файлах не создаёт
> дублей — приём считает по хешу содержимого. Тот же файл с новым содержимым заводит новую
> версию с пометкой «обновлённая», а не тихо подменяет старую.

## 5. Рабочий процесс проекта

Сквозной путь обследования. На каждом шаге вызывается свой навык; между шагами вы принимаете
результат. Навыки со статусом «в разработке» появятся в следующих версиях — их место в конвейере
уже зафиксировано.

| Этап | Навык | Что делает архитектор | Результат |
|---|---|---|---|
| **Разворот** | `new-project` | Ответить на вводный опрос, приложить ТКП/анкету, подтвердить каналы; по желанию — завести git/GitHub проекта. | Рабочее пространство и паспорт проекта. |
| **Аудит материалов** | `analyze-materials` | Проверить извлечённые факты и скелет процессов; спорное — уточнить. | Протокол аудита материалов. |
| **Разбор базы** *(опц.)* | `db-extract` | Задать выгрузки из базы заказчика как ещё один канал фактов. | Факты из системы в памяти. |
| **Планирование встреч** | `plan-meetings` | Утвердить план встреч и повестки перед отправкой заказчику. | Полный план/факт встреч + повестки. |
| **Встречи** | `analyze-meeting` | Подтвердить тип встречи; принять протокол; согласовать с заказчиком. | Типизированный протокол встречи. |
| **Синтез AS-IS** | `synthesis` | Судить спорные проблемы; проверить, что ущерб доказан, а не «на глаз». | Проблемы, ущерб, требования; книга памяти (xlsx). |
| **Решения** *(в разработке)* | `solutions` | Выбрать решения и продукты; развилки платформ — за заказчиком. | Решения и целевые продукты. |
| **Целевая архитектура** *(в разработке)* | `target-architecture` | Проверить целевую модель по уровням. | TO-BE по уровням каркаса. |
| **Дорожная карта** *(в разработке)* | `roadmap` | Утвердить волны и приоритеты. | Дорожная карта внедрения. |
| **Отчёт AS-IS** | `report-asis` | Проверить полноту; выдать заказчику. | Отчёт об обследовании AS-IS. |

## 6. Онбординг: что предложит new-project

Онбординг можно прервать и продолжить позже — прогресс хранится в файле `<проект>/.статус.yaml`;
если вы запустите любой навык плагина в незавершённом проекте, он напомнит о незакрытом шаге
и предложит его доделать.

В конце онбординга навык предлагает завести для проекта приватный репозиторий на GitHub —
по вашему согласию:

1. `git init` в папке проекта → первый коммит.
2. `gh repo create --private` — репозиторий создаётся строго приватным, для совместной работы
   команды.
3. Если утилиты `gh` нет — навык честно об этом сообщит и покажет команды для ручного выполнения.

**Что уходит в репозиторий:** всё содержимое проекта, кроме секретов — то есть память, узлы-
источники, документы, паспорт и правила, включая принятые файлы. Секреты (ключи API) в проекте
не хранятся вовсе (см. следующий раздел), поэтому физически утечь в git не могут.

**О чём навык предупреждает перед публикацией:**

- в репозитории будут **данные заказчика** — приватность репозитория обязательна, не понижайте
  видимость на публичную;
- **файлы больше 100 МБ** GitHub не примет — это лимит платформы; такие файлы придётся исключать
  или переносить отдельно.

Отказ от git/GitHub ничего не блокирует — это предложение, а не обязательный шаг.

## 7. Секреты: ключ Ньютона и другие API-ключи

Некоторые навыки обращаются к внешним сервисам (например, транскрибация записей встреч через
Ньютон) и для этого нужен API-ключ. Ключи хранятся **не в проекте**, а глобально у сотрудника —
в файле `~/.config/opp/secrets.env` (права на файл — 600, доступ только вам).

Это значит:

- ключ персональный, а не проектный — один ключ работает во всех ваших проектах;
- в папке проекта секретов нет вовсе, поэтому они не могут утечь в git проекта, даже случайно;
- при первом обращении к сервису, если ключа нет, навык сам спросит его у вас в диалоге и
  запишет в этот файл — вводить его руками или лезть в конфиги не нужно;
- если сервис ответил ошибкой авторизации (ключ отозван или введён неверно), навык переспросит
  ключ заново и перезапишет файл.

Формат файла простой (`KEY=VALUE` построчно), можно посмотреть или поправить его руками, но
обычно в этом нет необходимости — навык ведёт файл сам.

## 8. Архитектор и ИИ: кто что делает

**ИИ исполняет:** читает материалы и записи, чеканит факты с провенансом, ведёт дедупликацию,
собирает документы, проверяет целостность памяти. Он не выдумывает и честно помечает пробелы
(открытые вопросы).

**Архитектор судит:** разрешает спорное, задаёт границы понятий, принимает каждый гейт, отвечает
за стратегические развилки. Всё необратимое и выносимое заказчику решает человек, а не ИИ.

Правило: если факт не подтверждён или решение спорно — это открытый вопрос к вам, а не тихая
догадка в документе.

## 9. Выдача памяти: книга Excel и рабочие документы

После синтеза (и в любой момент дальше, по мере пополнения памяти) можно выгрузить всю
проектную память одной командой:

```
./opp export
```

Собирается книга **«Память проекта.xlsx»** в «Документы/»: титульный лист с названием проекта,
счётчиками и оглавлением с гиперссылками; на каждую непустую таблицу — отдельный лист (имя листа
= код таблицы, колонки — по контракту таблицы); служебная «история» изменений в книгу не
выводится; шапка каждого листа заморожена, включён автофильтр, строки — зеброй для читаемости.

**Книга — read-only представление.** Как и остальные документы продукта, она не редактируется
руками: если что-то нужно поправить, правка вносится в диалоге с навыком (он изменит факт в
памяти), после чего книга пересобирается командой `export` заново.

Отдельно от книги синтез также готовит два рабочих документа: **«Открытые вопросы»** и
**«Гипотезы»** — тоже как проекции памяти (docx), тоже пересобираемые, а не правимые вручную.

## 10. Справочник команд `./opp`

Основные команды. Навыки вызывают их сами; вручную они нужны для проверки и точечной работы.

| Команда | Что делает |
|---|---|
| `deploy` | Развернуть рабочую папку нового проекта (приёмные и технические папки, паспорт, правила). |
| `ingest` | Принять источники в проект (сырой слой) — идемпотентно, по хешу содержимого. |
| `extract` | Задание на извлечение строк конкретной таблицы из сырья проекта. |
| `framework` | Показать / перегенерировать описание каркаса (11 уровней) из данных. |
| `triangulate` | Свести факты по уровням, показать белые пятна — вход синтеза. |
| `prove` | Проверить доказательства ущерба: каждая проблема выше гипотезы имеет расчёт. |
| `requirements` | Отчёт по требованиям и покрытию проблем: непокрытые проблемы, требования без проблемы. |
| `solutions` | Отчёт по решениям (стадия осмысления). |
| `tobe` | Собрать целевую картину по уровням. |
| `roadmap` | Собрать дорожную карту (волны, приоритеты). |
| `render` | Собрать документ из памяти (протокол, отчёт, повестка, план встреч). |
| `split` | Вернуть согласованные правки из документа обратно в память. |
| `export` | Экспорт памяти проекта в xlsx («Память проекта.xlsx») — read-only представление. |
| `lint` | Проверить память проекта на целостность (0 ошибок — условие выдачи). |
| `verify` | Самопроверка продукта (для разработки плагина). |
| `status` | Сводка управляющего каталога проекта. |
| `ui` | Открыть окно проекта в браузере (по умолчанию порт 8765). |

## 11. Правила, которые нельзя нарушать

- **Не** выдумывать факты. Нет источника — это открытый вопрос, а не утверждение.
- **Не** править тело документа руками. Правится память, документ пересобирается.
- **Не** оставлять факт без провенанса — источник и место в нём обязательны.
- **Не** вести знание вне памяти. Проектная память — единый источник правды.
- **Всегда** запускать `./opp lint` перед любой выдачей — ноль ошибок обязателен.

## Приложение А. Словарь терминов

| Термин | Значение |
|---|---|
| Проектная память | Набор типизированных таблиц фактов — единственное место, где живёт знание о проекте. |
| Провенанс | Происхождение факта: ссылка на источник и точное место в нём (лист/раздел/таймкод). |
| Каркас (11 уровней) | Ось обследования: от ценности и учётных моделей до систем, интеграций и инфраструктуры. |
| Функциональный дефицит | Нехватка функции относительно эталона типового решения. |
| Теневое ИТ | Учёт и операции вне корпоративных систем (Excel, локальные базы). |
| Системная роль | Роль в системе с набором прав; отделена от организационной роли человека. |
| ОБЭ | Отрицательный бизнес-эффект — измеримый ущерб от проблемы (объём × ставка × частота). |
| Гейт | Точка приёмки: этап не считается закрытым, пока архитектор не принял результат. |
