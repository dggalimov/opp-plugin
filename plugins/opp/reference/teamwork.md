# Регламент командной работы OPP

Координация — только через **единый источник правды** (`plan.yaml` + `plan/00..06.yaml`).
Этот документ — правила игры.

## Действующий режим: одночатная финализация (ЗР-0021, с 03.07.2026)

Мультичатная разработка свёрнута решением владельца. Работает один участник — **«OPP Финал»**
(финализатор): исполняет задачи скоупа финала и готовит приёмку. **Приёмка — владелец лично**:
спеки утверждаются построчно; вехи фаз и переводы в «готово» — по слову владельца «принято»,
отметка — «ПРИНЯТО владельцем + дата» (R18). В силе: R9, R10, R11, R13, R17, R18. Неактивны
(один исполнитель): R1–R5, R7, R8, R12, R14–R16. Разделы ниже про роли и жизненный цикл
сохранены как канон мультичатного режима — на случай возврата к параллельной работе.

## Единый источник: один источник, два представления

- **Источник** — таблица элементов в `plan/<компонент>.yaml`. Элемент = атом продукта = единица работы.
- **Компонент** (00–06) — крупный бак (бывшая итерация). **Элемент** — реестр · инструмент · база-знаний ·
  навык · артефакт · профиль · внешний · интерфейс · проверка.
- Из таблицы выводятся два **представления** (`./opp`): «**Сборка продукта**» (архитектура по компонентам)
  и «**Доска**» (по статусу/исполнителю). Их не редактируют — правят строку элемента.
- Тяжёлые артефакты (контент БЗ, конфиги профилей, схемы таблиц) лежат файлами по полю `где`; таблица
  держит каталог + состояние, а не содержимое.

## Роли

- **OPP 00 — управляющий.** Не исполняет задачи. Планирует, ведёт таблицу и команду, проводит аудит
  стыков и приёмку. Единственный, кто переводит задачи в `бэклог`, `готово`, `отклонено`.
- **OPP NN — исполнитель.** Делает только назначенные ему задачи. Может предлагать новые (в статусе
  `валидация`).

## Жизненный цикл задачи

| статус | смысл | кто переводит сюда |
|---|---|---|
| `план` | в составе продукта, ещё не вытянут в работу | — (исходное) |
| `валидация` | предложен исполнителем, ждёт триажа; **работать нельзя** | любой исполнитель |
| `бэклог` | принят и назначен; будет делаться | **только OPP 00** |
| `в работе` | исполнитель начал (только свою из бэклога) | исполнитель |
| `проверка` | исполнитель завершил, ждёт приёмки | исполнитель |
| `готово` | OPP 00 проверил и принял | **только OPP 00** |
| `отклонено` | OPP 00 отклонил предложение (с причиной) | **только OPP 00** |

Маршруты: `план → бэклог` (OPP 00 вытянул и назначил) · `валидация → бэклог | отклонено` (триаж OPP 00) ·
`бэклог → в работе → проверка → готово`.

## Правила

- **R1.** OPP 00 не исполняет задачи — только план, аудит стыков, приёмка.
- **R2.** Исполнитель работает только над назначенной ему задачей `в работе` (взятой из бэклога). Неназначенное не берёт.
- **R3.** Исполнитель может добавить задачу — только в статусе `валидация`. В бэклог + назначение переводит только OPP 00.
- **R4.** Завершил → `проверка`. В `готово` переводит только OPP 00.
- **R5.** OPP 00 держит в плане, кто за что отвечает (исполнитель на каждой активной задаче).
- **R6.** Владелец объявляет имя чата → OPP 00 заносит его в команду.
- **R7. WIP = 1 на исполнителя.** Одновременно не больше одной задачи `в работе`. Делаем последовательно, не по пять сразу.
- **R8. Один файл — один исполнитель за раз.** OPP 00 при назначении следит, чтобы две задачи `в работе` не правили один файл/реестр (защита от конфликтов параллельных чатов).
- **R9. Стык — святое и направленный.** Меняешь `даёт`/`потребляет` — задача не уходит в `готово`, пока OPP 00 не сверил обе стороны.
- **R10. План — единственная синхронизация.** Сменил статус — сразу записал в `plan/<компонент>.yaml`. Нет записи = не произошло.
- **R11. Проверка целостности после каждого пополнения реестра.** Реестры растут на ходу; `./opp verify` зелёный — условие перевода в `готово`.
- **R12. Триаж не копится.** Каждую сверку OPP 00 разбирает `валидацию`: в бэклог (+назначение) или `отклонено` с причиной.
- **R13. Готово = проверяемо (DoD).** Перевод в `готово` только при: `./opp verify` зелёный + стык сверён + (для кода) тест/спека.
- **R18. Отметка приёмки (с 02.07).** Каждый перевод в `готово` сопровождается отметкой в `примечании` элемента: «**ПРИНЯТО OPP 00 + дата**»; в одночатном режиме (ЗР-0021) — «**ПРИНЯТО владельцем + дата**». Строка `готово` без такой отметки считается неподтверждённой. *Разовая амнистия 02.07:* 86 исторических строк, ставших `готово` до введения правила, считаются принятыми без отметки (внешний аудит 02.07 подтвердил целостность — `./opp verify` зелёный, 11 блоков, 150 тестов); правило обязательно для всех переводов **после** 02.07.

## Опознание и жизнь участника

- **R14. Опознание при старте.** При старте чата-исполнителя: если за ним не закреплено имя — чат **обязан
  попросить владельца объявить имя** и **не брать задачи**, пока OPP 00 не впишет его в команду.
- **R15. Закрепление имени.** Имя объявляет владелец; после объявления оно **закрепляется за чатом и не
  меняется**. OPP 00 фиксирует участника в команде со статусом `активен`.
- **R16. Архивирование.** Владелец может архивировать чат → OPP 00 ставит статус `архивирован`;
  незавершённые задачи этого чата (`в работе`/`проверка`) возвращаются в `бэклог` и переназначаются.
  Архивированный чат в работу не берётся.

## Архитектура: отчуждаемость

- **R17. Отчуждаемость (модульность).** Каждый элемент делается как самостоятельный отчуждаемый продукт:
  контракт наружу (`даёт`/`потребляет`), самодостаточная папка по `где`, зависимости только через
  `зависит_от`, никакого обхода стыка к внутренностям соседа. Навык/БЗ — пакуемый skill, готовый к выносу
  в отдельный плагин. `./opp verify` и аудит стыков OPP 00 это проверяют; нарушение — не уходит в `готово`.
  Несущий принцип — в `CLAUDE.md` («Отчуждаемость»).

## Проверка

`./opp verify` — целостность таблицы: существование путей `где`, согласованность `даёт`↔`потребляет`,
выводимые счётчики, отсутствие висячих зависимостей, **отчуждаемость** (фактические зависимости ⊆
`зависит_от`, нет обхода стыка, элемент в своей папке). Зелёный — обязательное условие приёмки (R13).
