# План разработки MCP для баз знаний OPP

> Самодостаточный документ для запуска разработки **в отдельном чате**. Содержит контекст,
> архитектуру MCP, контракты инструментов, технологические развилки, фазы и **шаблоны заданий**.
> При разработке используйте навык **anthropic-skills:mcp-builder** (лучшие практики MCP).

## 0. Контекст (для нового чата)

OPP (One Project Person) — плагин для ведения проекта внедрения 1С/ERP. У него есть **реестр баз
знаний** — строки типа «база-знаний» в `plan/03.yaml` продукта (59 БЗ; реестр растёт, вид — `knowledge/registry.py`): у каждой `id · название · тип · роль · домен · источник
· привязка · статус`.

- **Типы (технология):** `индексная` (структура 1С: метаданные/XML, точный lookup/diff) · `семантическая`
  (RAG по текстам: ИТС, CIMA, архив) · `плоская` (маленькая, в контексте) · `шаблоны` (образцы) ·
  `навигатор` (верхнеуровневая идентификация продукта).
- **Принцип:** операции OPP обращаются к БЗ **по `id`** через **единый интерфейс**; технология спрятана.
- Запросы локальных БЗ — офлайн (принцип: минимум зависимостей); исключение — базы с внешним источником (source.yaml «бэкенд: external-rag», напр. kb-bf-met → rag.1bitai.ru; контракт T-03-69, git-native).

Задача — собрать **MCP-сервер**, дающий OPP единый доступ к БЗ, и **конвейеры сборки**, которые
наполняют БЗ артефактами (индексы, векторные базы).

## 1. Архитектура

**Один MCP-сервер: `opp-knowledge`.** Читает реестр, маршрутизирует запрос к бэкенду по типу БЗ:

```
OPP / агент → MCP opp-knowledge → бэкенды:
                                   ├─ плоская      (вернуть текст)
                                   ├─ индексная    (lookup/diff по метаданным 1С)
                                   ├─ семантическая (поиск по векторному индексу)
                                   ├─ шаблоны      (список/выдача образцов)
                                   └─ навигатор    (определить продукт → id доменных БЗ)
```

**Сборка БЗ — отдельные конвейеры** (не часть MCP): на каждый тип — builder, который из источника
делает артефакт, который потом отдаёт MCP. Вызываются навыком «собрать базу знаний».

Артефакты БЗ: `knowledge/bases/<id>/` (индекс/векторная база/тексты). MCP читает их по `id`.

## 2. MCP `opp-knowledge` — контракт инструментов

| Инструмент | Вход | Выход | Назначение |
|---|---|---|---|
| `kb_list` | `filter?` (роль/тип/статус) | список БЗ (id, название, тип, роль, статус) | каталог |
| `kb_get` | `kb_id` | метаданные + статус сборки | карточка БЗ |
| `kb_query` | `kb_id`, `query`, `top_k?` | пассажи/записи + ссылки на источник | основной запрос (роутинг по типу) |
| `kb_navigate` | `описание_конфигурации` | предполагаемые продукты + id доменных БЗ | идентификация (навигатор) |
| `kb_build_status` | `kb_id?` | статус сборки (статусы ЖЦ таблицы элементов: план/…/готово, объём) | мониторинг |

Требования (по mcp-builder): понятные имена; типизированные входы/выходы; ошибки — как данные
(не исключения); каждый результат с **ссылкой на источник** (для доказуемости в OPP). `kb_query`
никогда не «выдумывает» — только то, что есть в БЗ.

## 3. Технологические развилки (решить в разработке)

1. **Семантический поиск (RAG):**
   - (а) **Лексический старт (BM25/keyword)** — без зависимостей, быстро, офлайн; хуже по смыслу.
   - (б) **Локальные эмбеддинги** (sentence-transformers / gguf через llama.cpp) + векторный индекс
     (sqlite/faiss/numpy) — лучше по смыслу; тяжелее.
   - **Рекомендация:** начать с (а) для проверки сквозного потока, перейти на (б) точечно.
2. **Индекс 1С (структура):** парсинг XML-выгрузки (`Configuration.xml` + метаданные) → структурированный
   JSON-индекс → точный lookup и diff с конфигурацией заказчика. Инструмент — парсер метаданных 1С.
3. **Хранение артефактов:** `knowledge/bases/<id>/` (текст/индекс/векторы) — git-native, по одной БЗ.

## 4. Фазы разработки

| Фаза | Что | Проверка |
|---|---|---|
| 1 | Каркас MCP (читает реестр; `kb_list`/`kb_get`/`kb_navigate` на плоских + навигаторе) | сервер поднимается, инструменты отвечают |
| 2 | Семантический бэкенд (RAG) + builder; собрать 1 БЗ (напр. `kb-audit` или `kb-cima`) | `kb_query` возвращает релевантные пассажи со ссылками |
| 3 | Индексный бэкенд (метаданные 1С) + builder; собрать 1 БЗ (напр. `kb-erp-str` из XML) | точный lookup/diff по объекту конфигурации |
| 4 | Подключение к OPP: единый запрос по id + привязки к процедурам заполнения реестров | OPP получает подсказки из БЗ с провенансом-подсказки (≠ факт) |

## 5. Шаблоны заданий на разработку

> Скопируйте нужный шаблон в новый чат, заполните плейсхолдеры `<…>`. Начинайте с навыка

### Шаблон A — Каркас MCP `opp-knowledge`
```
Контекст: см. knowledge/mcp-development-plan.md (разделы 0–2). Реестр БЗ — строки «база-знаний» в plan/03.yaml (вид — knowledge/registry.py).
Задача: реализовать MCP-сервер opp-knowledge (фаза 1).
- Инструменты: kb_list, kb_get, kb_navigate (по плоским БЗ и навигатору).
- Сервер читает реестр; технология запроса спрятана за единым интерфейсом.
Контракт/приёмка: сервер поднимается; инструменты типизированы; ошибки как данные; каждый ответ
с ссылкой на источник; есть тесты на 3 инструмента.
Инструменты разработки: навык anthropic-skills:mcp-builder.
Куда: <папка MCP в продукте OPP или отдельный репозиторий>.
```

### Шаблон B — Семантический бэкенд (RAG)
```
Контекст: knowledge/mcp-development-plan.md (раздел 3.1). Тип БЗ: семантическая.
Задача (фаза 2): реализовать RAG-бэкенд и builder.
- Развилка: начать с лексического (BM25) офлайн; заложить переход на локальные эмбеддинги.
- builder: источник текста (файлы/выгрузка) → разбиение → индекс в knowledge/bases/<id>/.
- kb_query(kb_id, query, top_k) → пассажи + ссылки на источник.
Приёмка: на тестовой БЗ (<kb-audit/kb-cima>) запрос возвращает релевантное со ссылками; повторяемо.
```

### Шаблон C — Индексный бэкенд (метаданные 1С)
```
Контекст: knowledge/mcp-development-plan.md (раздел 3.2). Тип БЗ: индексная.
Задача (фаза 3): парсер XML-выгрузки типовой 1С → структурированный индекс + lookup/diff.
- Вход: XML-выгрузка конфигурации (Configuration.xml + метаданные).
- Выход: индекс объектов (справочники, документы, регистры, отчёты, роли…) в knowledge/bases/<id>/.
- kb_query поддерживает точный lookup и diff «типовая vs заказчик» (функциональный дефицит).
Приёмка: на тестовой БЗ (<kb-erp-str>) точный поиск объекта и сравнение с заказчиком.
```

### Шаблон D — Собрать одну БЗ
```
Контекст: plan/03.yaml — БЗ <id> типа <тип>, источник <источник>.
Задача: собрать БЗ <id>.
- Взять источник (<файлы/ссылка/выгрузка>), применить builder типа <тип>.
- Сохранить артефакт в knowledge/bases/<id>/; в реестре статус <id> → «готова».
Приёмка: kb_query(<id>, <тест-запрос>) возвращает корректный результат со ссылкой; ./opp verify зелёный.
```

## 6. Чего НЕ делать в этом плане
- Живой коннектор к конфигурации заказчика (mcp-1c) — это **инструмент извлечения данных по
  источникам**, не БЗ. Обсуждается в итерации «Источники».
- Не тащить тяжёлые внешние сервисы в ядро — тяжёлые бэкенды подключаются точечно по типу БЗ.
