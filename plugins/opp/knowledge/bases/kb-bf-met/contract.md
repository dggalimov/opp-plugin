# kb-bf-met — контракт базы (БИТ.Финанс — методика, внешний RAG)

**Роль:** доменная-методика. **Домен:** БИТ.Финанс. **Тип:** семантическая (внешний RAG).
**Источник:** руководство пользователя БИТ.Финанс через наш внешний RAG `rag.1bitai.ru`.
**Особенность:** локального корпуса НЕТ (в отличие от ИТС-баз) → репозиторий не пухнет (нет проблемы 679 МБ).

## Что база даёт (стык наружу)
- На `kb_query("kb-bf-met", вопрос)` — **ранжированные пассажи** из руководства БИТ.Финанс **со ссылкой на источник** (файл · чанк).
- Привязка к каркасу: бюджетирование, казначейство, управление договорами, управленческий учёт/МСФО, консолидация (Ур.1, 2, 3; разнесение типовой↔надстройка — Ур.8).
- **Доказуемость:** ответ — только из руководства; ничего не выдумывать. Нет релевантных пассажей → **ошибка-данные** «не найдено», а не сочинённый ответ.
- Генерации нет: RAG `result=null` — это ретривер; база отдаёт пассажи, а не готовый текст. Свод/ответ собирает потребитель поверх пассажей.

## Как работает (маршрут запроса)
```
kb_query("kb-bf-met", вопрос)
   → core: тип «семантическая», бэкенд = внешний-rag (по source.yaml)
   → внешний-RAG бэкенд MCP (строит OPP 02): POST rag.1bitai.ru/api/v1/collections/<коллекция>/query {query, top_k}
   → context_docs → пассажи {текст, источник: «руководство БИТ.Финанс · чанк N», оценка}
```
Машинные параметры (эндпоинт, коллекция, маппинг ответа, токен-из-секрета) — в [`source.yaml`](source.yaml).

## Стык с OPP 02 (согласован OPP 00 30.06, контракт — в примечании T-03-69)
- **Признак маршрутизации:** `source.yaml`, поле `бэкенд: external-rag | local-bm25`; нет файла → local-bm25 (нужен `chunks.jsonl`).
- **Подключение:** `endpoint` + `collection` + `token_env` — плоские поля `source.yaml` (их читает `backends/source_config.py`). Для rag.1bitai.ru — `OPP_RAG_1BITAI_TOKEN` (env либо `~/.config/opp/secrets.env`); сам токен владелец объявил несекретным и он продублирован в `source.yaml` (02.07.2026).
- Формат пассажа на выход `kb_query` — единый с локальным семантическим (`{текст, источник, оценка}`), чтобы потребители не различали бэкенды.
- **Факт живого прогона (02.07.2026, аудит):** эндпоинт отвечает; в пассаже `score` лежит в `metadata`, а на верхнем уровне только `content`+`metadata` — защитному маппингу бэкенда (`external_rag._passages`) нужно читать `metadata.score` и собирать «источник» из константы провенанса + `metadata.chunk_index` (ключей `source/citation/metadata.title` в ответе НЕТ, `original_filename` = техимя «result_json.txt»). Коллекция — 1677 чанков.

## Эталонность (важно для плана)
Это **единственная** БИТ.Финанс-методичка. 5 комбо `-bf-met` (ERP/КА/УТ/БП/БП+стройка) НЕ дублируют БИТ.Финанс: комбо = **базовый-ИТС (`kb-<base>-met`) + эта `kb-bf-met`**. Навигатор и комбо-маршрутизацию надо привести к этому (правка стыка → OPP 00): «надстройка БИТ.Финанс → добавить kb-bf-met к базовой методичке», а не отдельная объединённая база.
