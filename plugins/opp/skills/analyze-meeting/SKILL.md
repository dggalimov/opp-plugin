# Навык: Аудит через интервью

> Это навык ПРОДУКТА (репозиторий OnePersonProject). Одноимённый навык старого плагина
> (~/.claude/plugins/cache/opp/) — только справка, ему не следовать.

Суждение поверх рельсов: читаешь транскрипт встречи, выносишь услышанное в типизированные
строки таблиц с провенансом до таймкода, собираешь протокол на согласование. Контракт полей,
инструкции и валидацию несут рельсы (`"${CLAUDE_PLUGIN_ROOT}/opp" extract`, `write_rows`) — не дублируй; профили
встреч — реестр `capabilities/extract/meeting-types.yaml` (единый источник, не дублируй).

## Несущие правила (методика kb-audit; нарушение = брак)

1. **Провенанс прежде строки.** Поле «источник» = код SRC узла записи, «локатор в
   источнике» = таймкод `[ЧЧ:ММ:СС]` или диапазон (из diarize-JSON со спикерами; он
   приоритетнее плоского .txt). Транскрипт без таймкодов — локатор по фрагменту реплики
   + отметка в MTG/примечании. Нет якоря — нет строки; локатор обязателен у каждой строки.
2. **Статус — «гипотеза», и не выше.** Встреча — один канал (методика р.4). Повышение
   статуса доказанности — работа синтеза (правило двух каналов), НЕ этого навыка:
   поле «статус доказанности» существующих строк не трогай.
3. **Слова — не система.** Природа доказательства — «слова»: заявление на встрече ≠
   подтверждённое состояние системы; надёжность — по роли говорящего (владелец процесса
   о своём процессе ≠ мнение о чужом). Не выдумывать: догадка → HYP/OQ, не каркасная таблица.
4. **«И что из этого?»** Наблюдение без следствия — фактура таблиц уровня; кандидат в
   проблему (HYP) — только когда виден потенциальный ущерб или расхождение.
5. **Количества — только в VP** (C7): любое прозвучавшее число — строка VP со своим
   источником-таймкодом; другие таблицы ссылаются, не держат чисел.
6. **Дедуп до чеканки.** Задание (`"${CLAUDE_PLUGIN_ROOT}/opp" extract`) отдаёт «уже_в_памяти»: на встречах ты
   чаще ПОДТВЕРЖДАЕШЬ уже известное — пиши тот же код, write_rows сольёт и накопит источники
   списком. Новый код — только для нового факта, слагом по словарю
   `${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/словарь-слагов.md`.
7. **Расхождение канала — объект.** Услышанное противоречит материалам (или другой встрече)
   → HYP со ссылками на ОБА источника. Не сглаживай.
8. **Институциональный проход.** После извлечения фактов — отдельный проход по транскрипту:
   регламентные СРОКИ, ПОРОГИ и допуски, ОРГАНЫ и лестницы утверждения, САНКЦИИ и запреты.
   На встречах их проговаривают вскользь («ну это как обычно к 11-му дню…») — каждое такое
   правило — ОТДЕЛЬНАЯ строка BR («одно правило источника = одна строка»).
9. **Договорённости и поручения → OQ**: кто, что обещал/запросил, «что нужно, чтобы закрыть»,
   ответственный. Отдельной таблицы поручений нет — не выдумывай.
10. **Повторный прогон не переизвлекает молча.** Встреча с MTG «проведена» и уже собранным
    протоколом — обработана; повторный вызов навыка на ней предлагает пересборку протокола
    (тот же md → docx), а не запускает извлечение фактов заново (см. Шаг 1).

## Процедура

### Шаг 0. Приём записи

а. **Вложения чата.** Если пользователь прислал запись вложением, а не файлом в проекте —
   сначала скопируй файл в приёмную папку `<проект>/Встречи/` (это архив, откуда идёт приём;
   работай дальше с копией там, не с временным вложением).

б. **Ключ Ньютона.**
   ```python
   from capabilities.extract.secrets import get_secret
   ```
   `get_secret("NEWTON_TOKEN")` — каскад: переменная окружения → `~/.config/opp/secrets.env`.
   Нет ключа → спроси у пользователя (он лежит у сотрудника глобально, не в проекте) →
   `save_secret("NEWTON_TOKEN", "<значение>")` → продолжай тем же прогоном. Транскрибация
   вернула ошибку авторизации от newton (протух/неверный ключ) → перезапроси у пользователя
   и `save_secret` заново (перезапись), затем повтори транскрибацию.
   Запись — видео, а ffmpeg не установлен («нужен ffmpeg») → предложи установить и, ПОСЛЕ
   явного согласия пользователя, выполни `brew install ffmpeg` (нет Homebrew — покажи команду
   его пакетного менеджера) и повтори приём. Отказался — запись остаётся без транскрипта,
   зафиксируй причину в реестре.

в. **Приём.**
   ```
   "${CLAUDE_PLUGIN_ROOT}/opp" ingest "<проект>/Встречи" --into "<проект>"
   ```
   Идемпотентен: уже принятые записи (по хешу содержимого) пропускаются ДО транскрибации —
   Ньютон повторно не тратится; тот же путь с новым содержимым — новый узел с пометкой
   «обновлённая версия». Покажи пользователю отчёт команды как есть (новых / уже были /
   обновлений, по каждой записи — распознана/нет).

### Шаг 1. Защита от повторного прогона

Для каждого узла записи (SRC-REC), который предстоит разбирать в этом прогоне (новые из
отчёта Шага 0в или явно указанные пользователем), проверь ДО начала разбора: есть ли уже
строка **MTG** со статусом «проведена», ссылающаяся на этот источник (`источники из встречи`),
И собран ли уже протокол по нему — идемпотентная проверка идёт по **md-канону** (техслой),
не по клиентскому docx (у него теперь человеческое имя формулой `имя_файла`, не пригодное для
поиска по коду): существует ли `<проект>/Проектная память/Документы/meeting-<SRC-код>.md`.
Оба условия — встреча уже обработана: сообщи пользователю и предложи пересборку протокола
(повтор Шага 6 без нового прохода по транскрипту), НЕ переизвлекай факты молча. Ни одно из
условий не выполнено — узел новый для разбора, переходи к Шагу 2.

### Шаг 2. Контекст, профиль встречи и кадры

Прочитай паспорт проекта. Определи тип встречи: frontmatter узла записи (`тип_встречи`
после ingest) или по содержанию; профили и фокус — `capabilities/extract/meeting-types.yaml`.
**Подтверди тип у пользователя, не решай молча.** Профиль задаёт приоритетные таблицы:
демо-систем → SYS/CR/GAP/INT (+ кадры); процессы → PRC/BR/ORG; стейкхолдеры → REP/OQ;
оргвстреча → MTG/OQ; прочее — по содержанию.

Тип «демо-систем» и запись — видео (не только аудио) → предложи пользователю «разобрать на
кадры?». Согласился →
```python
from capabilities.extract.meetings import frames_dir, extract_frames
```
`frames_dir(workspace, "<SRC-REC-код>")` даёт (и создаёт) папку
`Проектная память/Источники/_кадры/<код>/`; `extract_frames(video_path, out_dir)` извлекает
кадры в неё. Путь для протокола в Шаге 6 — относительный от корня проекта:
`Проектная память/Источники/_кадры/<код>`.

Если у встречи есть повестка (строки AGD по коду MTG — заведены навыком планирования встреч),
прочитай её и держи пункты как чек-лист разбора: повестка задаёт, что планировали выяснить.

### Шаг 3. Сопоставление с планом (перед строкой MTG)

Прежде чем завести строку MTG «проведена», поищи среди существующих строк MTG со статусом
«запланирована» ту, что относится к этой встрече — по теме, дате и участникам (суждение,
не точное совпадение; неоднозначно — спроси пользователя, не гадай). Ориентиры: дайджест
`"${CLAUDE_PLUGIN_ROOT}/opp" extract MTG --into "<проект>"` (коды и темы уже заведённых встреч) и, при необходимости
сверки по дате/участникам, чтение `<проект>/memory/MTG.yaml` (только для чтения — само
сопоставление; запись всё равно идёт через `write_rows`, см. ниже).

- **Совпала** → тот же код MTG: статус «запланирована» → «проведена», проставь фактическую
  дату и участников (план→факт), не заводи новый код.
- **Не нашлась** → НОВАЯ строка MTG «проведена» с примечанием «вне плана».

Заведи (если ingest не завёл) строку **SRC** узла записи: тип канала «Люди», природа
доказательства «слова», дата встречи, ссылка на встречу (MTG).

Обработав так ВСЕ новые встречи этого прогона (Шаги 1–3 для каждой), выведи пользователю
**план/факт В СООБЩЕНИИ** — markdown-таблица (код, тема, дата, статус) — с подсветкой
**жирным**: какие строки обработаны сейчас (совпали с планом), какие добавлены вне плана.
Дождись подтверждения пользователя, прежде чем переходить к Шагу 4 (глубокий разбор —
дорогая работа, её не стоит переделывать из-за неверного сопоставления). По желанию
пользователя пересобери docx полного план/факта: `"${CLAUDE_PLUGIN_ROOT}/opp" render meeting-plan --into "<проект>" --docx`.

### Шаг 4. Проход по транскрипту — трёхтактный протокол полноты (спека 08 §4)

Та же дисциплина, что у материалов (`opp:analyze-materials`): выборочное чтение без описи —
брак. Разбор ведётся тремя тактами вокруг одной строки **MAT**, заведённой на саму запись
(тип материала «запись встречи», канал поступления «передан на встрече», ссылка на источник —
код SRC-REC этой записи) — если строки ещё нет, заведи её в начале такта 1.

**Такт 1 — опись (до извлечения).** По грубой карте тем (см. п.1 такта 2) посчитай, сколько
сущностей каких каркасных таблиц транскрипт заявляет предметно — сколько процессов, правил,
систем и т.п. называется по существу, не мимоходом (дёшево даже для длинной записи). Запиши
YAML-словарём в поле MAT «опись сущностей» (например `{PRC: 8, BR: 3, SYS: 2}`); статус разбора
строки MAT — «в разборе».

**Такт 2 — извлечение.**
1. Читай diarize-JSON (спикеры + секунды), если есть; иначе .md/.txt узла. **Большой
   транскрипт (>200 КБ)** — чанками по таймкодам: сначала грубая карта тем (кто о чём в
   какие минуты — она же основа описи такта 1), затем плотное чтение приоритетных отрезков
   по профилю.
2. Для каждой затронутой таблицы возьми задание: `"${CLAUDE_PLUGIN_ROOT}/opp" extract <КОД> --into "<проект>"`.
3. Извлекай по несущим правилам: якорь-таймкод → строка; числа → VP; совпадение с
   «уже_в_памяти» → тот же код; противоречие → HYP; поручение → OQ.
4. Запиши через рельсы (см. «Запись строк»). ERROR валидации — почини строку, не обходи.
Порядок таблиц (MTG уже заведена в Шаге 3): VP → приоритетные таблицы профиля → остальные
затронутые → HYP/OQ.

**Такт 3 — счётная сверка.** Сверь число извлечённых строк по каждой таблице описи с
заявленным в такте 1. Расхождение чинится (дочитай пропущенное по транскрипту) или честно
фиксируется: статус разбора строки MAT — «частично разобран» + открытый вопрос (OQ) на
дочитку с числом остатка (например «PRC: разобрано 5 из 8»). Полное покрытие описи — статус
разбора «разобран». Статус «частично разобран» без описи и без OQ на дочитку не допускается —
это брак протокола. `"${CLAUDE_PLUGIN_ROOT}/opp" lint` в Шаге 6 механически сверяет опись MAT
с фактически извлечённым и подсвечивает WARN «недоизвлечено», если сверка выше пропущена или
ошибочна.

### Шаг 5. Пробелы встречи и судьбы пунктов повестки

Что планировали выяснить, но не выяснили (тема заявлена, ответа нет) — OQ с критичностью
и ответственным. Если была повестка (AGD) — каждому пункту проставь судьбу через рельсы:
«обсуждён» + итог одной-двумя фразами (факты — не в итог, а в каркас через источник),
«перенесён» (на какую встречу — в примечание) или «снят» (почему — в примечание).
Заявленный необсуждённый пункт = кандидат в OQ, не потеря.

### Шаг 6. Самопроверка и протокол

- `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR.
- Собери протокол на согласование:
  `"${CLAUDE_PLUGIN_ROOT}/opp" render meeting --into "<проект>" --источник <SRC-код> [--кадры Проектная память/Источники/_кадры/<код>] --docx`
  (`--кадры` — только для демо-встреч с разобранными кадрами, путь из Шага 2).
- Обнови обзор проекта (`обзор.md`): встреча разобрана, строк добавлено/подтверждено,
  открытые вопросы.

### Шаг 7. После согласования заказчиком

Правленый md протокола → `"${CLAUDE_PLUGIN_ROOT}/opp" split meeting "<файл>" --into "<проект>"`. Редактируемые
колонки: MTG.статус, MTG.участники, OQ.статус закрытия, OQ.резолюция, статус HYP, AGD.статус,
AGD.итог обсуждения. Остальные правки split отклонит предупреждением — это норма: память
меняется фактами, не редактурой.

Различай характер правки:
- **Структурная** (состав секций, оформление, порядок) → спроси «запомнить как шаблон
  проекта?»; согласился — запиши/поправь `<проект>/«шаблоны документов»/meeting.yaml`
  (замещает продуктовый конфиг целиком для этого документа в этом проекте) и сделай пробный
  рендер (`"${CLAUDE_PLUGIN_ROOT}/opp" render meeting --into "<проект>" --источник <SRC-код> --docx`), чтобы
  убедиться, что шаблон валиден.
- **Фактическая** (содержание, цифры, формулировки) → в память (write_rows / split выше) →
  пересборка md → docx.

## Запись строк (рельсы)

Строки пишутся ТОЛЬКО через `schema.extract.write_rows` (валидация + слияние + история),
не редактированием `memory/*.yaml` руками:

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<PY
import sys; sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")
import json
from schema.extract import write_rows
problems = write_rows("<КОД>", json.load(open("rows.json", encoding="utf-8")), "<проект>")
for p in problems: print(p)
PY
```

## Выход

Отчёт приёма (Шаг 0в); строка MTG по встрече, сопоставленная с планом или помеченная «вне
плана» (Шаг 3); строка MAT на запись с описью сущностей и честным статусом разбора (Шаг 4,
трёхтактный протокол); план/факт с подсветкой, показанный и подтверждённый пользователем
(Шаг 3); строки таблиц с якорями «SRC@таймкод» (новые факты + подтверждения существующих
вторым источником); HYP по расхождениям каналов; OQ по поручениям, пробелам и недочитанному
остатку описи; кадры демо-встречи (если согласился пользователь); судьбы пунктов повестки
(AGD: статус + итог), если повестка была; lint 0 ERROR (WARN «недоизвлечено» — сигнал, не
блокер); протокол встречи (md+docx, структура — по типу встречи, движок ветвится сам) собран
на согласование; обзор обновлён.

## Границы

НЕ транскрибирует заново, если транскрипт есть (transcribe — рельса для живых записей);
НЕ повышает статусы доказанности и НЕ триангулирует (синтез); НЕ синтезирует PRB/EFF/TB;
НЕ распознаёт кадры (OCR нет — кадры вставляются картинками в протокол); НЕ правит чужие
строки иначе как слиянием через write_rows; НЕ переизвлекает факты по встрече, уже имеющей
MTG «проведена» и протокол (Шаг 1) — только пересборка по запросу; НЕ рассылает и НЕ создаёт
повестки (planning — skills/plan-meetings), только проставляет судьбы уже заведённых пунктов.
