# Навык: Планирование встреч

Суждение поверх рельсов: из живой **Картины каркаса** собираешь план встреч с повестками — не
из сырого списка вопросов, а из осмысления дыр всех 11 уровней. Навык ПРЕДЛАГАЕТ — владелец
утверждает план перед рассылкой повесток заказчику. Контракт полей и валидацию несут рельсы
(`"${CLAUDE_PLUGIN_ROOT}/opp" picture`, `"${CLAUDE_PLUGIN_ROOT}/opp" extract`, `write_rows`); типы встреч — реестр
`capabilities/extract/meeting-types.yaml` (единый источник, не дублируй).

## Несущие правила

1. **Цель проекта — фокус глубины, НЕ фильтр полноты (спека 08 §3а.1).** Каркас закрывается
   комплексно ВСЕГДА, по всем 11 уровням. Цель проекта (например, миграция) определяет, где
   копать глубже и что ставить в приоритет ближайших встреч, — но НЕ отменяет покрытие
   остальных уровней. План встреч закрывает дыры каркаса в целом, а не только тематику цели;
   если картина показывает «не начат»/«в разведке» на уровнях вне цели — эти дыры идут в план
   наравне с целевыми. Уход повесток только в тему цели при куче дыр каркаса — брак планирования.
2. **Повестка рождается из картины, не из фантазии.** Вход навыка — свежая Картина каркаса
   (шаг 0), а не сырой список OQ. Три источника пунктов на каждом уровне: (а) **гипотезы-эскизы
   уровня** на подтверждение (шаг 1), (б) **открытые вопросы уровня** (OQ картины — дыры,
   ставшие вопросами), (в) незакрытые обязательные каналы сценария эксплуатации (канал людей
   без проведённой встречи). Пункт из OQ несёт ссылку «происхождение-вопрос» — провенанс
   планирования; пункт из гипотезы формулируется как «мы предполагаем X — так ли это?».
3. **Гипотеза-эскиз пропорциональна знанию (спека 08 §3а.3).** Большое белое пятно (уровень/
   контур почти не покрыт) → гипотеза УКРУПНЁННАЯ: состав контура, ключевые процессы,
   владельцы — БЕЗ операционных деталей (детали на большом пятне слишком гипотетичны, это
   брак). Зона со сформированным контекстом → детальная гипотеза (конкретный шаг, правило,
   стык). Цель встречи — устранить противоречия и пробелы, мешающие сформулировать полноценный
   каркас, а не только уточнить точечные детали.
4. **Стадии погружения — от общего к частному.** Встречи группируются по стадиям С1–С4
   (см. правило и шаг 2); дыра/гипотеза уровня → тема повестки встречи СВОЕЙ стадии. Стадия не
   перескакивается при незакрытых уровнях предыдущей без явного решения владельца (фиксируй в
   примечании MTG). Это защищает «сначала скелет, потом детали».
5. **Тип встречи — по содержанию пунктов.** Пункты про механику процессов → «процессы-asis»;
   про систему/доработки/демо → «демо-систем»; про цели и ценность → «стейкхолдеры»;
   организационные → «оргвстреча». Не смешивай разнотипные пункты в одну встречу без нужды —
   тип задаёт и профиль разбора, и структуру будущего протокола.
6. **Реалистичный объём.** 3–10 пунктов на встречу (один слот интервью); критичные вопросы и
   гипотезы ранних стадий — в ближайшие встречи. Не дробить до микровопросов — детализация
   живёт в OQ.
7. **Участники — под пункты.** Подбирай участников (ссылки на STK) по владению темами пунктов
   (владелец процесса — о своём процессе; ИТ-владелец — о системах на С4); нет нужного
   стейкхолдера в реестре — пометь пункт «нужен носитель темы» в примечании, не выдумывай людей.
8. **Вводные и подготовка участников — зачем встреча и что принести.** Поле MTG «вводные»: цель
   встречи и контекст 2–4 фразами (что уже знаем, что хотим закрыть) — в шапку повестки. Поле
   MTG «подготовка участников»: что продемонстрировать/принести (демонстрация процесса/системы,
   файлы, выгрузки) — в раздел «Пожелания к участникам» повестки.
9. **Не трогай проведённое.** Навык создаёт и правит только строки статуса «запланирована» и их
   пункты; проведённые встречи и их повестки — зона навыка analyze-meeting.

## Процедура

### Шаг 0. Свежая картина (вход)
Прогони картину ПЕРВЫМ действием — она вход навыка, не сырой список вопросов:
`"${CLAUDE_PLUGIN_ROOT}/opp" picture --into "<проект>" --вопросы`. Флаг `--вопросы` дозаводит OQ из непокрытых дыр
(белое пятно, с уровнем; дедуп по существующим). Прочитай канон
`<проект>/Проектная память/Рабочие/картина.md`: по каждому из 11 уровней — собрано фактов,
пройден ли ведущий канал, дыры полноты, вопросы уровня (OQ/HYP), статус готовности
(«не начат / в разведке / готов к синтезу / синтезирован»). Прочитай паспорт проекта и сценарий
эксплуатации (обязательные каналы людей без проведённой встречи). Пожелания владельца к
обсуждению — спроси. Триггер актуализации шире, чем «после встреч»: запускай и когда с прошлого
плана появились новые материалы (картина пересчитается сама — новые дыры видны).

### Шаг 1. Гипотезирование (проактивные эскизы на крупные дыры)
По КРУПНЫМ дырам картины (уровень «не начат»/«в разведке», контур почти не покрыт) построй
гипотезы-эскизы комплексных процессов/контуров. Опирайся на три основания ПО ПОРЯДКУ
(спека 08 §3а.2):
1. **Отраслевые лучшие практики** — доменные БЗ продукта по профилю заказчика (провенанс —
   узел-источник природы «эталон», SRC-KB → KBP); нет подключённой БЗ — общеотраслевое знание
   с ЯВНОЙ пометкой об этом в формулировке гипотезы.
2. **Сформированный контекст каркаса** заказчика — что уже знаем из материалов/встреч (картина).
3. **Рамки и цели проекта** — паспорт.

Эскиз утверждает, КАК ДОЛЖЕН быть устроен контур у такого заказчика («судя по отрасли, контур
бюджетирования включает планирование → исполнение → контроль; проверить состав и владельцев»).
Гранулярность — по правилу 3 (большое пятно → укрупнённо; знакомая зона → детально).
Запись через рельсы: возьми задание `"${CLAUDE_PLUGIN_ROOT}/opp" extract HYP --into "<проект>"` (контракт, занятые
коды, дедуп) и заведи строку HYP: `вид гипотезы` = **«эскиз по отраслевым практикам»**; `уровень`
(обязателен — уровень контура); `формулировка` (укрупнённая или детальная по знанию); `что
проверить` = какие встречи/вопросы подтверждают эскиз; `статус проверки` = «открыта»; источник
и провенанс — как задаёт контракт (SRC-KB / общеотраслевое с пометкой). Пиши через `write_rows`
(см. «Запись строк»); коды — смысловые слаги по словарю
(`${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/словарь-слагов.md`).

### Шаг 2. Раскладка по стадиям и скелет плана
Разложи дыры и гипотезы уровней по **стадиям погружения** (спека 08 §3), от общего к частному:
- **С1 — общая картина и ценность** (Ур.1–2; стейкхолдеры/руководство): для чего бизнес, где
  создаётся ценность.
- **С2 — функции и процессы обзорно** (Ур.3–5, скелет; носители контуров, владельцы —
  оргструктура Ур.6 как носитель контуров цепляется сюда): состав контуров, ключевые процессы
  и правила обзорно.
- **С3 — углублённые процессы** (Ур.5 детально, по одному контуру за встречу): механика шагов,
  точки решения, стыки внутри контура.
- **С4 — системы и ИТ** (Ур.7–11; ИТ-владельцы): системы, доработки, интеграции, роли,
  инфраструктура.

Стадия не перескакивается при незакрытых уровнях предыдущей без явного решения владельца —
если приходится (приоритет цели), зафиксируй причину в примечании MTG. Сгруппируй пункты одной
стадии по носителям знания и типам встреч (правило 5). На каждую группу — строка MTG: код
(слаг по словарю), тема (заголовок), тип, статус «запланирована», планируемая дата (согласуй
окно с владельцем), участники (STK), вводные, подготовка участников. Пиши через рельсы
(`write_rows`).

### Шаг 3. Повестки
На каждую запланированную встречу — пункты AGD (порядок, формулировка, происхождение-вопрос,
статус «заявлен»). Повестка несёт три вида пунктов (правило 2):
- **гипотезы уровня на подтверждение** — формулировка «Мы предполагаем, что … — так ли это?
  (что проверить: состав/владельцы/правило)»; провенанс эскиза остаётся в HYP «что проверить»;
- **открытые вопросы уровня** — формулировка вопросом («Как сейчас…?», «Кто вправе…?»),
  `происхождение-вопрос` = код OQ;
- **пожелания к подготовке** — в поле MTG «подготовка участников» встречи (не отдельный AGD).

Критичные OQ и гипотезы ранних стадий не должны остаться вне повесток ближайших встреч — проверь.

### Шаг 4. Самопроверка и документы
- `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR.
- План встреч — полный план/факт (спека 06-f): `"${CLAUDE_PLUGIN_ROOT}/opp" render meeting-plan --into "<проект>" --docx`
  собирает три секции по статусу («Запланированные» / «Проведённые» / «Отменённые», пустая
  скрывается) + повестки + открытые вопросы к закрытию. Показывай владельцу весь документ, не
  только новые строки, — это единый обзор план/факта проекта.
- Повестка каждой встречи: `"${CLAUDE_PLUGIN_ROOT}/opp" render agenda --into "<проект>" --встреча <код MTG> --docx`
  (шапка без титульной страницы → пункты повестки → «Пожелания к участникам»).
- Покажи владельцу план и повестки — **рассылка заказчику только после его утверждения**
  (рассылка — вне продукта, владелец отправляет сам).

### Шаг 5. Актуализация
После проведения встреч (навык analyze-meeting проставит статусы/итоги пунктов) картина
пересчитается сама; пересобери план этим же навыком с новой картины: незакрытые пункты
«перенесён» — в повестки следующих встреч новой строкой; подтверждённые гипотезы уходят из
повесток (перенесены в каркас); новые дыры — новые пункты и гипотезы.

### Шаг 6. Шаблон проекта (правки структуры)
Владелец правит структуру/оформление плана или повестки (не факты, а разделы/колонки/
формулировки шапки) → спроси «запомнить как шаблон проекта?» → по согласию запиши
`<проект>/шаблоны документов/meeting-plan.yaml` или `agenda.yaml` (замещает продуктовый конфиг
целиком, `capabilities/render/engine.py::load_config`) → сделай пробный рендер
(`"${CLAUDE_PLUGIN_ROOT}/opp" render meeting-plan --into "<проект>" --docx` или `agenda …`), чтобы поймать ошибку
шаблона (проекция/дубли номеров секций) сразу.

## Запись строк (рельсы)

Строки MTG/AGD/HYP пишутся ТОЛЬКО через `schema.extract.write_rows` (валидация + слияние +
история), не редактированием `memory/*.yaml` руками:

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<PY
import sys; sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")
import json
from schema.extract import write_rows
problems = write_rows("<КОД>", json.load(open("rows.json", encoding="utf-8")), "<проект>")
for p in problems: print(p)
PY
```

ERROR в ответе — строка не записана: исправь по сообщению и повтори, не обходя правило.

## Выход

Свежая картина (канон + дозаведённые OQ из дыр); гипотезы-эскизы HYP «эскиз по отраслевым
практикам» на крупные дыры (укрупнённо/детально по знанию); строки MTG «запланирована»
разложены по стадиям С1–С4, с вводными и подготовкой участников; пункты AGD (гипотезы на
подтверждение + OQ уровня); lint 0 ERROR; «План встреч» — полный план/факт (md+docx) и повестки
встреч (md+docx) на утверждение владельцу.

## Границы

НЕ проводит и НЕ разбирает встречи (analyze-meeting); НЕ закрывает OQ и НЕ подтверждает гипотезы
(закрытие/подтверждение — по факту ответа на встрече, через протокол/каналы); НЕ рассылает
повестки заказчику (владелец, вне продукта); НЕ правит проведённые встречи и их повестки; НЕ
создаёт стейкхолдеров под выдуманных людей; НЕ пишет каркасные строки уровней (это каналы —
эскизы живут в HYP до подтверждения).
