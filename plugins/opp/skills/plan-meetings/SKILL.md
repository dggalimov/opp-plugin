# Навык: Планирование встреч

Суждение поверх рельсов: из накопленных дыр разведки собираешь план встреч с повестками.
Навык ПРЕДЛАГАЕТ — владелец утверждает план перед рассылкой повесток заказчику. Контракт
полей и валидацию несут рельсы (`write_rows`); типы встреч — реестр
`capabilities/extract/meeting-types.yaml` (единый источник, не дублируй).

## Несущие правила

1. **Повестка рождается из дыр, не из фантазии.** Три источника пунктов: открытые вопросы
   (OQ, статус «открыт», в первую очередь критичные), незакрытые обязательные каналы
   сценария эксплуатации (этап 0 — канал людей без проведённой встречи), белые пятна
   триангуляции (`"${CLAUDE_PLUGIN_ROOT}/opp" triangulate` — уровни каркаса без покрытия/с одним каналом).
   Пункт из OQ несёт ссылку «происхождение-вопрос» — провенанс планирования.
2. **Тип встречи — по содержанию пунктов.** Пункты про механику процессов → «процессы-asis»;
   про систему/доработки/демо → «демо-систем»; про цели и ценность → «стейкхолдеры»;
   организационные → «оргвстреча». Не смешивай разнотипные пункты в одну встречу без нужды —
   тип задаёт и профиль разбора, и структуру будущего протокола.
3. **Реалистичный объём.** 3–10 пунктов на встречу (один слот интервью); критичные вопросы —
   в ближайшие встречи. Не дробить до микровопросов — детализация живёт в OQ.
4. **Участники — под пункты.** Подбирай участников (ссылки на STK) по владению темами
   пунктов (владелец процесса — о своём процессе); нет нужного стейкхолдера в реестре —
   пометь пункт «нужен носитель темы» в примечании, не выдумывай людей.
5. **Вводные — зачем встреча.** Поле MTG «вводные»: цель встречи и контекст 2–4 фразами
   (что уже знаем, что хотим закрыть) — идёт в шапку повестки и протокола.
6. **Не трогай проведённое.** Навык создаёт и правит только строки статуса «запланирована»
   и их пункты; проведённые встречи и их повестки — зона навыка analyze-meeting.

## Процедура

### Шаг 0. Сбор дыр
Триггер актуализации — шире, чем «после проведения встреч» (спека 06-f, фаза 3): запускай
и когда с прошлого плана появились НОВЫЕ материалы (новые узлы MAT, новые OQ из их разбора) —
не только после встреч. Прочитай паспорт проекта и сценарий эксплуатации (обязательные каналы
людей). Собери: `"${CLAUDE_PLUGIN_ROOT}/opp" triangulate --into "<проект>"` (белые пятна по уровням, в т.ч. новые —
от свежих материалов), открытые OQ (критичность, «что нужно, чтобы закрыть», ответственный,
включая появившиеся после последнего плана), обязательные каналы без проведённых встреч
(MTG: нет строки «проведена» по каналу). Пожелания владельца к обсуждению — спроси.

### Шаг 1. Скелет плана
Сгруппируй дыры по носителям знания и типам встреч (правило 2). На каждую группу — строка
MTG: код (слаг по словарю), тема (заголовок), тип, статус «запланирована», планируемая дата
(согласуй с владельцем окно), участники (STK), вводные. Пиши через рельсы (`write_rows`,
как в навыке analyze-meeting).

### Шаг 2. Повестки
На каждую запланированную встречу — пункты AGD: порядок, формулировка вопросом
(«Как сейчас…?», «Кто вправе…?»), происхождение-вопрос (код OQ, если пункт из вопроса),
статус «заявлен». Критичные OQ не должны остаться вне повесток ближайших встреч — проверь.

### Шаг 3. Самопроверка и документы
- `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR.
- План встреч — теперь ПОЛНЫЙ план/факт (спека 06-f): `"${CLAUDE_PLUGIN_ROOT}/opp" render meeting-plan --into
  "<проект>" --docx` собирает три секции по статусу («Запланированные» / «Проведённые» /
  «Отменённые», пустая скрывается) + повестки + открытые вопросы к закрытию. Показывай
  владельцу весь документ, не только новые запланированные строки, — это единый обзор
  план/факта проекта, не только то, что добавил этот прогон.
- Повестка каждой встречи: `"${CLAUDE_PLUGIN_ROOT}/opp" render agenda --into "<проект>" --встреча <код MTG> --docx`.
- Покажи владельцу план и повестки — **рассылка заказчику только после его утверждения**
  (рассылка — вне продукта, владелец отправляет сам).

### Шаг 4. Актуализация
После проведения встреч (навык analyze-meeting проставит статусы/итоги пунктов) план
пересобирается этим же навыком: незакрытые пункты «перенесён» — в повестки следующих
встреч новой строкой; новые дыры — новые пункты.

### Шаг 5. Шаблон проекта (правки структуры)
Владелец правит структуру/оформление плана или повестки (не факты, а разделы/колонки/
формулировки шапки) → спроси «запомнить как шаблон проекта?» → по согласию запиши
`<проект>/шаблоны документов/meeting-plan.yaml` или `agenda.yaml` (замещает продуктовый
конфиг целиком, `capabilities/render/engine.py::load_config`) → сделай пробный рендер
(`"${CLAUDE_PLUGIN_ROOT}/opp" render meeting-plan --into "<проект>" --docx` или `agenda …`), чтобы поймать
ошибку шаблона (проекция/дубли номеров секций) сразу, не в следующем прогоне.

## Выход

Строки MTG «запланирована» с вводными и участниками; пункты AGD с происхождением из OQ;
lint 0 ERROR; «План встреч» — полный план/факт (md+docx: запланированные/проведённые/
отменённые + повестки + открытые вопросы) — и повестки встреч (md+docx) собраны на
утверждение владельцу.

## Границы

НЕ проводит и НЕ разбирает встречи (analyze-meeting); НЕ закрывает OQ (закрытие — по факту
ответа, через протокол/каналы); НЕ рассылает повестки заказчику (владелец, вне продукта);
НЕ правит проведённые встречи и их повестки; НЕ создаёт стейкхолдеров под выдуманных людей.
