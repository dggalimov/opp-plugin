# Навык: Аудит материалов

> Это навык ПРОДУКТА (репозиторий OnePersonProject). Одноимённый навык старого плагина
> (~/.claude/plugins/cache/opp/) — только справка, ему не следовать.

Суждение поверх рельсов: читаешь узлы-источники, выносишь факты в типизированные строки таблиц
с провенансом до фрагмента. Контракт полей, инструкции и валидацию несут рельсы (`"${CLAUDE_PLUGIN_ROOT}/opp" extract`,
`write_rows`) — не дублируй их; твоя работа — прочесть, понять, привязать и не выдумать.

## Несущие правила (методика kb-audit; нарушение = брак)

1. **Провенанс прежде строки.** Сначала найди цитату/фрагмент в узле, потом пиши строку:
   поле «источник» = код SRC узла, поле «локатор в источнике» = страница/раздел/лист/строка.
   Нет якоря — нет строки. **Локатор обязателен у КАЖДОЙ строки с источником** (кроме SRC/MAT):
   перед записью пройди по строкам батча — пустой «локатор в источнике» = строка не готова
   (гейт Ф1 показал: без этой проверки локатор теряется в 85% строк).
2. **Статус — «гипотеза».** Материалы — один канал (методика р.4): всё, что извлечено здесь,
   получает «статус доказанности: гипотеза». Не выше. Повышают статусы другие каналы.
3. **Не выдумывать.** Факт — только то, что узел утверждает. Догадка «наверное у них…» —
   в HYP (гипотеза-расхождение) или OQ (открытый вопрос), не в каркасную таблицу.
4. **«И что из этого?»** Наблюдение без следствия для бизнеса — фактура таблиц уровня;
   кандидат в проблему (HYP) заводи только когда виден потенциальный ущерб или расхождение.
5. **Количества — только в VP** (C7): любое число (объём, частота, штат, документооборот) —
   строка VP со своим источником; другие таблицы ссылаются на неё, не держат чисел.
6. **Дедуп до чеканки.** Задание (`"${CLAUDE_PLUGIN_ROOT}/opp" extract`) отдаёт «уже_в_памяти» (занятые коды +
   формулировки) и строки предшественников: сперва ищи существующую строку — при совпадении
   пиши тот же код (write_rows сольёт, история сохранится), новый код — только для нового факта.
   **Коды — смысловые слаги** (`AM-BDDS`, `BR-KPI-FCAST`, `ORG-CFO`), НЕ порядковые (`AM-01`):
   порядковый код у другого прогона другой — ломается дедуп и повторяемость.
7. **Расхождение — объект.** Материал противоречит материалу → HYP со ссылками на оба узла;
   вопрос без ответа в материалах → OQ. Не сглаживай.
8. **Институциональный проход (итерация гейта Ф1).** После извлечения фактов узла — отдельный
   проход по «институциональным правилам»: регламентные СРОКИ (дни, периодичность, «на N-й день»),
   ПОРОГИ и допуски (%, суммы, лимиты), ОРГАНЫ и лестницы утверждения (кто/в какой последовательности),
   САНКЦИИ и запреты («нет X — нет Y»). Каждое такое правило — ОТДЕЛЬНАЯ строка BR
   («одно правило источника = одна строка», не обобщать: три вида служебных записок = три строки).
   В больших узлах они прячутся в сносках, примечаниях, листах «вопросы»/«правила»/«свод» —
   гейт показал: агрегатное чтение их глотает.
9. **Чеканка слагов — по словарю.** Канонический словарь сокращений и правила слагов:
   `${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/словарь-слагов.md`. Отступление от
   словаря ломает дедуп и повторяемость (гейт: пересечение кодов 32% при смысловом 64%).

## Процедура

### Шаг 0а. Приём вложений чата (архив)
Материалы пришли вложениями в диалог, а не файлами в папке проекта? Сначала скопируй их в
приёмную папку `<проект>/Входные материалы/` — это архив пользователя; из временных путей
чата напрямую ничего не обрабатывается. Дальше — только через неё, наравне с тем, что
пользователь положил сам.

### Шаг 0б. Приём: `"${CLAUDE_PLUGIN_ROOT}/opp" ingest` и блокирующий гейт
`"${CLAUDE_PLUGIN_ROOT}/opp" ingest "<проект>/Входные материалы" --into "<проект>"` — идемпотентный приём (дедуп по
sha256-хешу содержимого; тот же файл повторно → «уже был»; тот же путь с новым содержимым →
новый узел с пометкой «обновлённая версия»). Покажи пользователю отчёт: **новых N — СПИСКОМ
с названиями файлов**, уже были M, обновлений K. Дальше — **блокирующий гейт**: к шагу 1
переходишь только после явного согласия пользователя; он может уточнить состав или попросить
исключить часть новых узлов из разбора этого прогона (файлы уже приняты как SRC — исключение
из разбора не отменяет приём, только откладывает его).

### Шаг 1. Контекст проекта
Прочитай паспорт проекта и профиль конфигурации (если есть). Открой чек-лист таблиц:
`${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/чек-лист-таблиц.md` — что искать
в материалах для каждой таблицы (выведен из framework.yaml).

### Шаг 2. Инвентаризация (полнота входа — 100%, без исключений)
Собери список ВСЕХ узлов `Источники/**/*.md` (рекурсивно). Для каждого узла:
- строка **SRC** (если её ещё нет): код узла из frontmatter, название, тип канала, дата,
  способ получения, природа доказательства («документ» для материалов), надёжность,
  статус обработки;
- строка **MAT**: ссылка на SRC, статус разбора («разобран» / «не поддаётся разбору» + причина).
Нечитаемый узел не пропускается молча — MAT-строка со статусом и OQ при необходимости.
Сверь счёт: строк MAT = узлов. Не сходится — найди потерянный узел.
**Повторный прогон навыка на том же проекте**: узел, у которого уже есть строка MAT со
статусом «разобран», в разбор (шаг 3) заново не берётся — только новые и обновлённые узлы
(обновлённая версия ingest даёт новый SRC-код, значит и новую MAT — разбирается как новый).
Сверка счёта MAT=узлов остаётся проверкой ПОЛНОТЫ реестра, а не сигналом «разобрать всё заново».

### Шаг 3. Проход по узлам (батчами 10–15)
Для каждого узла в батче:
1. Прочитай узел целиком. **Большой узел (>200 КБ, обычно Excel-выгрузка):** сначала структура
   (frontmatter, заголовки листов `## `, шапки таблиц, счёт строк), затем выборочное чтение
   репрезентативных фрагментов; количества фиксируй в VP по агрегатам (число строк листа,
   диапазоны), НЕ читая всё; в MAT статус «частично разобран», перечень листов и причина —
   в примечании (тот же статус — узлу, потерявшему содержимое при конвертации).
2. Определи, какие уровни каркаса узел затрагивает (чек-лист).
3. Для каждой затронутой таблицы возьми задание: `"${CLAUDE_PLUGIN_ROOT}/opp" extract <КОД> --into "<проект>"` —
   в нём контракт полей, эффективная инструкция, занятые коды, строки предшественников.
4. Извлеки строки по несущим правилам (якорь → строка; числа → VP; сомнение уровня → HYP/OQ).
5. Запиши через рельсы (см. «Запись строк» ниже). ERROR валидации — почини строку по
   сообщению, не обходя правило.
Порядок таблиц внутри узла: MAT → VP → REP → AM → BR → NSI → PRC → ORG → SYS → CR → GAP →
INT → KBP → HYP/OQ. Для PRC — по 20/80: до узлов процессов; шаги — только где материал
реально даёт карточку шага.

### Шаг 4. Пробелы → запросы
Уровень каркаса, по которому материалов нет вовсе, — это не «пусто», это белое пятно:
заведи OQ и подготовь запрос заказчику по шаблону
`${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/шаблон-запроса-недостающего.md`.

### Шаг 5. Самопроверка и обзор
- `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR (висячие ссылки, дубли, словари, инварианты).
- Обнови обзор проекта (файл `обзор.md` в корне воркспейса — его создаёт deploy): что
  заполнено (строк по таблицам), белые пятна по уровням, следующий шаг.
- Сверь ещё раз M1: строк MAT = узлов.

### Шаг 6. Финал: саммари или протокол
Спроси пользователя, что нужно:
- **саммари в чате** — живой ответ по памяти о том, что дали материалы (не документ:
  свободное саммари несовместимо с повторяемостью — принцип ЗР-0025 п.4); можно выгрузить
  тем же текстом в md-файл без статуса документа, если пользователь попросит файл;
- **«Протокол аудита материалов»** — детерминированный документ:
  `"${CLAUDE_PLUGIN_ROOT}/opp" render materials --into "<проект>" --docx`. Ссылку на docx отдай пользователю.

Пользователь просит поправить протокол после рендера:
- **оформление** (шрифты, темы, вид) — правится в «Оформление.yaml» проекта, не здесь;
- **структура** (состав/порядок секций, номера, заголовки) — правь через конфиг документа
  и предложи «запомнить как шаблон проекта»: по согласию запиши
  `<проект>/шаблоны документов/materials.yaml` — полную копию эффективного конфига
  (продуктового `documents/materials/config.yaml`, если это первая правка) с внесёнными
  правками. Проектный шаблон целиком замещает продуктовый конфиг для этого документа во
  всех последующих рендерах (движок: `capabilities/render/engine.py::load_config`). Сразу
  прогони пробный `"${CLAUDE_PLUGIN_ROOT}/opp" render materials --into "<проект>"` — это валидация: битый шаблон
  (нарушена проекция секции или задвоен номер) даёт понятную ошибку `RenderError` — почини
  шаблон, не обходи. После успешной пробы — пересобери docx (`--docx`) для пользователя.

## Запись строк (рельсы)

Строки пишутся ТОЛЬКО через `schema.extract.write_rows` (валидация + слияние + история),
не редактированием `memory/*.yaml` руками:

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<PY
import sys; sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")
import json
from schema.extract import write_rows
problems = write_rows("<КОД>", json.load(open("rows.json", encoding="utf-8")), "<проект>")
for p in problems: print(p)
PY
```

ERROR в ответе — строка не записана: исправь и повтори. WARN («поле вне контракта») — проверь
имя поля по заданию.

## Выход

Материалы приняты идемпотентно (отчёт ingest, гейт пройден с согласия пользователя);
наполненные `memory/*.yaml` по содержимому НОВЫХ/обновлённых материалов; MAT = 100% узлов
реестра; каждая строка — с источником и локатором; lint 0 ERROR; обзор проекта обновлён;
список запросов заказчику; саммари в чате или «Протокол аудита материалов» (md+docx) —
по выбору пользователя; правки структуры протокола, при согласии, — в проектном шаблоне
`<проект>/шаблоны документов/materials.yaml`.

## Границы

НЕ синтезирует проблемы/требования (PRB/EFF/TB — навык синтеза после интервью); НЕ меняет
узлы-источники; НЕ правит чужие строки иначе как слиянием через write_rows; НЕ заполняет
таблицы вне профиля пилота без явного основания в материале.
