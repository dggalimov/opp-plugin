# Навык: Синтез AS-IS

> Это навык ПРОДУКТА (репозиторий OnePersonProject). Одноимённые навыки старого плагина —
> только справка, им не следовать.

Суждение сведения: читаешь триангуляцию памяти, сводишь сигналы в проблемы, доказываешь ущерб,
выводишь требования. Структурную плоскость несут рельсы (`"${CLAUDE_PLUGIN_ROOT}/opp" triangulate`, `"${CLAUDE_PLUGIN_ROOT}/opp" lint`,
`"${CLAUDE_PLUGIN_ROOT}/opp" prove`, `"${CLAUDE_PLUGIN_ROOT}/opp" requirements`) — семантику (что считать одним корнем, где ущерб, какое
требование отвечает проблеме) судишь ты.

## Несущие правила (методика kb-audit; нарушение = брак)

1. **Только синтез пишет PRB/EFF/TB.** Навыки каналов дают сигналы — ты сводишь.
   Никакая PRB не рождается без сигнала в памяти: у каждой проблемы — источник(и) и якорь.
2. **Дедуп по корню — но корень это ЯВЛЕНИЕ, не ТЕМА.** Одно явление, увиденное в N сигналах,
   — ОДНА проблема; все якоря — в источники строки (write_rows накопит). Проверка границы:
   два кандидата — один корень, ЕСЛИ устраняются одним изменением; лечатся по-разному —
   это РАЗНЫЕ проблемы (обычно каскадные к общему корню: «бюджет в Excel» — корень, а
   «правила аллокации захардкожены сотнями строк», «зависимость от внешней BI-службы»,
   «дробление баз из-за лимита строк» — самостоятельные каскадные PRB со своим ущербом).
   Не размножай одинаковое — и не сплющивай разное: свёртка разных механизмов в одну
   строку-тему теряет проблемы так же, как размножение плодит дубли (гейт-2: укрупнение
   до тем стоило 39 пунктов recall).
3. **Статус — по каналам (методика р.4).** «гипотеза» — сигнал одного канала; «интервью» —
   независимые якоря материала И встречи (69 фактов памяти уже двухканальные — кандидаты);
   «цифровой след» — только при данных системы (выгрузка/регистр, не слова); «согласовано» —
   только после согласования заказчиком (не в этом прогоне). Повышение статуса тянет гейт ОБЭ:
   без категории + «в чём ущерб» + EFF-ссылки линт не пропустит — и это правильно.
4. **Доказательство без выдумки (C13).** EFF собирается ТОЛЬКО из множителей, существующих в
   памяти: «объём» и «частота» — ссылки на VP (со своими источниками), «ставка» — число из
   источника, «способ расчёта» — словами. Величина ущерба = произведение множителей
   (`"${CLAUDE_PLUGIN_ROOT}/opp" prove` пересчитает — расхождение будет дефектом). Нет множителей в источниках —
   величина НЕ пишется: EFF со способом расчёта и пустой величиной + OQ «запросить данные».
   Оценок «на глаз» не бывает; правдоподобие величины судит владелец, не ты.
5. **Расхождение каналов — в проблему.** Материал говорит одно, встреча — другое: фиксируй
   в «расхождение-источник» строки PRB или отдельной PRB о недостоверности данных. Не сглаживай.
6. **Требование = бизнес-результат.** TB: действие + объект/функция + критерий приёмки +
   бизнес-результат + «устраняемый ОБЭ» (ссылка на EFF). Каждая PRB с ущербом покрыта хотя бы
   в одном из двух полей покрытия TB («покрывает проблемы (полностью)» / «...(частично)»,
   проблема не может стоять в обоих списках сразу); требований без проблемы не бывает.
   Формулируй результатом («факт УУ доступен на 3-й день»), не решением («внедрить модуль X»
   — это Ф5).
7. **Нижняя отсечка.** Наблюдение без ущерба и без расхождения — не проблема: помечай полем
   «нижняя отсечка» / оставляй фактурой уровня. Тип решения не выдумывай — он следует из
   уровня (методологическое Ур.1–3 / организационное Ур.4–6 / технологическое Ур.7–11, C6).
8. **Сигнальная полнота (итерация гейта-2).** После чеканки пройди ВСЕ сигналы памяти
   поимённо: каждую HYP обоих каналов; каждую дефектную строку каркаса (рассинхрон/дубли/
   грязь/теневое ИТ у NSI, ручные шаги и разрывы PRC, техдолг и незапущенные CR,
   институциональные BR с видимым риском, разрывы INT, перегруз SYS); каждое расхождение
   триангуляции. Судьба КАЖДОГО сигнала — одна из трёх, и она записывается в фиксированный
   файл карты синтеза `<проект>/Проектная память/Рабочие/карта-синтеза.md`:
   (а) вошёл в PRB (источники/расхождение-источник), (б) отсечён с причиной (нижняя
   отсечка / TO-BE-замысел / нет ущерба), (в) OQ — не хватает данных судить. Молчаливо
   потерянных сигналов НЕТ; карта синтеза — доказательство полноты, её сверяет гейт.
   Повторный запуск синтеза в проекте: файл уже существует — прочти его и ДОПОЛНЯЙ (новые
   сигналы с прошлого прогона), не переигрывай уже записанные судьбы заново.
9. **Синтез — от полной картины; цель не сужает каркас (спека 08 §1, §3а.1).** Синтез
   начинается с Картины каркаса и НЕ стартует на непокрытом каркасе: уровень со статусом
   «не начат» или критичный открытый OQ — стоп (шаг 0, гейт по картине). Полноту считай по
   ВСЕМ 11 уровням независимо от цели проекта: цель (например, миграция) задаёт глубину и
   приоритет, но не выкидывает уровни — синтезируется весь каркас, а не только тематика цели.
   Обход гейта — только явным допущением владельца, оно фиксируется в OQ «закрыт допущением».

## Процедура

### Шаг −1. Методика (первый запуск)
Прочти `<проект>/.статус.yaml`. Ключ «методика-синтеза-показана: false» (или ключа нет —
проект развёрнут до появления этого шага) → первый запуск синтеза в проекте: покажи
владельцу выжимку методики синтеза — несущие правила этого навыка (выше) + отсылка к
`knowledge/bases/kb-audit/методика.md` разделы 5–7 («Перекрёстная сверка», «Диагностика
проблемы — четыре координаты», «От проблемы к требованию») — прими уточнения владельца
(если есть) → запиши их в `<проект>/Правила проекта.md`, раздел «## Уточнения методики
(проект)» (создай раздел, если его ещё нет; сама kb-audit не правится — единый источник
методики) → поставь `методика-синтеза-показана: true` в `.статус.yaml`. Уже true — шаг
пропускается молча. В ЛЮБОЙ сессии по прямому запросу владельца «покажи методику» —
та же выжимка + текущие проектные уточнения из «Правила проекта.md» (независимо от
маркера).

### Шаг 0. Картина и гейт синтеза
Синтез начинается с картины (правило 9). Прогони `"${CLAUDE_PLUGIN_ROOT}/opp" picture --into "<проект>"` и прочитай
канон `<проект>/Проектная память/Рабочие/картина.md`: статус готовности каждого из 11 уровней и
критичные открытые OQ. **Гейт:** есть уровень со статусом **«не начат»** ИЛИ критичный
(«стоп перед синтезом») открытый OQ → **СТОП**: покажи владельцу картину («какие уровни не
покрыты, какие критичные вопросы висят») и не чекань PRB/EFF/TB, пока каркас не доведён
разведкой. Обход — только явным допущением владельца по конкретному уровню/вопросу: зафиксируй
его как OQ со статусом «закрыт допущением» (резолюция — текст допущения), и тогда синтез идёт
дальше. Полнота считается по ВСЕМ уровням независимо от цели проекта — не пропускай нецелевые
уровни (правило 9).

Гейт пройден (или снят допущениями) → `"${CLAUDE_PLUGIN_ROOT}/opp" triangulate --into "<проект>"` — строки по
источникам и уровням, перекрёстные сверки, белые пятна. Прочитай `обзор.md` (заметки каналов:
повреждённые записи, условные привязки — спорные факты перепроверяй по узлам-источникам).

### Шаг 1. Карта корней
Собери сигналы: все HYP (двух каналов), каркасные строки с зафиксированными дефектами
(рассинхрон НСИ, теневое ИТ, ручные шаги PRC, техдолг CR), расхождения триангуляции.
Сгруппируй по корню (правило 2): временная карта «корень → сигналы → уровни → якоря».
TO-BE-замыслы (встречи-проектирование) — НЕ проблемы AS-IS, отсекай.

### Шаг 2. Чеканка PRB
Для каждого корня возьми задание `"${CLAUDE_PLUGIN_ROOT}/opp" extract PRB --into "<проект>"` (контракт, занятые
коды, предшественники) и заведи строку: формулировка; координата 1 — уровень + каркасная
строка; координата 2 — диагноз (из словаря дефектов уровня); координата 3 — ОБЭ (категория,
«в чём ущерб»; «сколько» — после EFF); координата 4 — тип решения (по уровню); каскад
(«корневая проблема» у следствий); статус по правилу 3; источник + локатор (ведущий якорь,
остальные накопятся). Коды — смысловые слаги
(`${CLAUDE_PLUGIN_ROOT}/skills/analyze-materials/assets/словарь-слагов.md`).

### Шаг 3. Доказательство (EFF)
Для каждой PRB выше «гипотезы» (и где память позволяет — для остальных): строка EFF по
правилу 4. Ссылку «координата 3 — ОБЭ: сколько» проставь в PRB. Прогони `"${CLAUDE_PLUGIN_ROOT}/opp" prove` —
дефекты доказательств чини по сообщению, не обходя правило.

### Шаг 4. Требования (TB с покрытием проблем)
Из проблем — требования по правилу 6; в каждой строке TB заполни покрытие: коды проблем в
«покрывает проблемы (полностью)» и/или «покрывает проблемы (частично)» — проблема не может
быть в обоих списках сразу. Прогони `"${CLAUDE_PLUGIN_ROOT}/opp" requirements` — «непокрытые проблемы» и «требования
без проблемы» разбери до нуля или объясни в обзоре (например, проблема-констатация без
требования — редкость, обоснуй).

### Шаг 5. Самопроверка
- `"${CLAUDE_PLUGIN_ROOT}/opp" lint "<проект>"` — 0 ERROR (гейт ОБЭ, замкнутость PRB×TB, ацикличность, тип↔уровень);
- `"${CLAUDE_PLUGIN_ROOT}/opp" prove` и `"${CLAUDE_PLUGIN_ROOT}/opp" requirements` — чисто или объяснено;
- обнови `обзор.md`: сколько корней, сколько выше гипотезы, покрытие требованиями,
  нижняя отсечка, что ждёт данных заказчика;
- сверь карту синтеза (`Проектная память/Рабочие/карта-синтеза.md`, правило 8) — у каждого
  пройденного сигнала есть судьба, молчаливых пропаж нет.

### Финал. Выдачи
После зелёного шага 5 — таблицы памяти в презентационные документы (спека 06-f, фаза 5):
- `"${CLAUDE_PLUGIN_ROOT}/opp" export --into "<проект>"` — книга «Память проекта.xlsx» в «Документы/» (титул,
  счётчики, лист на непустую таблицу; read-only представление);
- `"${CLAUDE_PLUGIN_ROOT}/opp" render oq --into "<проект>" --docx` — «Открытые вопросы»;
- `"${CLAUDE_PLUGIN_ROOT}/opp" render hyp --into "<проект>" --docx` — «Гипотезы».

Дай владельцу ссылки на все три файла. Напомни: xlsx и docx — read-only представления
памяти; правки — через диалог → память → пересборка тем же навыком, не руками в файле.
Команда синтеза запускается многократно по мере пополнения памяти — финал повторяется
каждый раз (документы пересобираются из текущей памяти, не накапливаются версиями).

## Запись строк (рельсы)

Только через `schema.extract.write_rows` (валидация + слияние + история):

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<PY
import sys; sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")
import json
from schema.extract import write_rows
problems = write_rows("<КОД>", json.load(open("rows.json", encoding="utf-8")), "<проект>")
for p in problems: print(p)
PY
```

## Выход

Реестр PRB (дедуплицирован по корню, каскад, статусы честные) + EFF (доказательства без
выдумки) + TB (полное покрытие проблем полями «покрывает проблемы»); lint 0 ERROR;
prove/requirements чисто; обзор обновлён; карта синтеза ведёт судьбу каждого сигнала;
выдачи владельцу — «Память проекта.xlsx», «Открытые вопросы.docx», «Гипотезы.docx».

## Границы

НЕ подбирает решения и продукты (Ф5); НЕ строит TO-BE и дорожную карту (Ф5); НЕ правит
строки каналов иначе как слиянием; НЕ считает деньги вне произведения множителей (КРП — вне
финала); НЕ поднимает статус без двух каналов; НЕ пишет величину ущерба без множителей.
