# Навык: Запуск нового проекта (онбординг)

Провести пользователя по семи шагам онбординга: структура → паспорт → карточка компании →
правила → оформление → гайд и git/GitHub → завершение. Прогресс — в `<проект>/.статус.yaml`;
онбординг возобновляемый — прерванный запуск продолжается с прерванного шага, не с начала.

## Шаг 0. Проверь, не продолжение ли это
Если у проекта уже есть `.статус.yaml` и `онбординг.завершён: false` при `онбординг.шаг > 0` —
онбординг начат раньше и прерван. Скажи пользователю «продолжаю онбординг с шага N» и перейди
сразу к этому шагу ниже, не повторяя пройденные. Если `.статус.yaml` нет или `шаг: 0` — начни
с шага 1.

## Шаг 1. Разворот структуры
Пользователь даёт родительскую папку; имя по умолчанию — `<Заказчик> — OPP`. Спроси кратко,
какие каналы данных ожидаются (документы / база данных / интервью) — не знает пользователь,
оставь все три. Подтверди и разверни:

```bash
"${CLAUDE_PLUGIN_ROOT}/opp" deploy "<родитель>/<Заказчик> — OPP" --channels "документы,интервью"
```

Развёртывание идемпотентно и создаёт: приёмные **«Входные материалы/»** и **«Встречи/»** —
сюда пользователь кладёт свои файлы; технический слой **«Источники/»** — туда пользователь
не ходит, это узлы-источники после разбора; «Проектная память/», «Документы/»; паспорт,
правила и оформление по умолчанию; `.gitignore`; `.статус.yaml`. Повтор не затирает заполненное.
Папки «00 — Управление проектом» в структуре больше нет. → отметь `онбординг.шаг: 1`.

## Шаг 2. Опрос паспорта
Прими и опрос в чате, и документы (`$ARGUMENTS` или брошенные файлы: ТКП, анкета, любые
вводные). Неструктурированный вход разбери сам; **вопросы можно пропускать** — не держи
пользователя на пустых полях. Собери под состав паспорта
(`${CLAUDE_PLUGIN_ROOT}/reference/passport.template.md`): клиент, задача, цель и критерии
успеха, задача выбора, аудируемые блоки, фокус-уровни, глубина и каналы сценария, команды,
системы, объёмные параметры, рамки. Заполни `<проект>/Паспорт проекта.md` (правь файл или
попроси заполнить в чате). → `онбординг.шаг: 2`.

## Шаг 3. Карточка компании
Запусти навык `company-card` на этом же проекте (Skill company-card). Он сам: соберёт публичные
факты, сохранит узел `SRC-COM` в «Источники/» **и выведет карточку сжатым текстом в чат**
пользователю на сверку. Дождись подтверждения или правок пользователя; правки вносятся в
`SRC-COM` (навык company-card это умеет). → `онбординг.шаг: 3`.

## Шаг 4. Правила проекта (активный короткий опрос)
Покажи пользователю `<проект>/Правила проекта.md` — он уже заполнен значениями по умолчанию
(`reference/principles.default.md`). Задай несколько коротких вопросов по ключевым пунктам,
устраивают ли значения по умолчанию:
- **гейт синтеза** — когда считать разведку достаточной, чтобы запускать синтез;
- **статусная лестница** — устраивает ли «гипотеза → подтверждено цифровым следом →
  подтверждено интервью → согласовано заказчиком», или нужна другая;
- **язык и тон** документов (по умолчанию — русский, экспертный).

Правки пользователя вноси прямо в файл (не в память — это настройка, не факт). → `онбординг.шаг: 4`.

## Шаг 5. Оформление
Сообщи: «Оформление документов по умолчанию — корпоративный стиль Первый Бит (уже
активировано). Если нужно оформление под бренд заказчика — отдельный навык `theme-from-brand`
(понадобятся логотип, сайт заказчика или примеры документов)». Сам навык `theme-from-brand`
не вызывай — это решение и отдельный запуск пользователя. → `онбординг.шаг: 5`.

## Шаг 6. Финал: гайд и git/GitHub
Скопируй гайд пользователя из `${CLAUDE_PLUGIN_ROOT}/reference/guide/` в корень папки проекта
(все файлы, без изменений) — это готовый asset продукта, «Руководство пользователя».

Предложи создать репозиторий: «Хотите вести проект в git/GitHub — для командной работы?
Репозиторий будет **строго приватным** (в нём данные заказчика); файлы крупнее 100 МБ GitHub
отклонит — учтите при добавлении тяжёлых записей». По согласию пользователя:

```bash
cd "<проект>" && git init && git add -A && git commit -m "Онбординг проекта"
```

Проверь, установлен ли `gh`:

```bash
command -v gh
```

Есть `gh` — создай приватный репозиторий и опубликуй:

```bash
gh repo create --private --source=. --remote=origin --push
```

Нет `gh` — честно скажи пользователю, что утилита не установлена, и дай команды для ручного
создания: завести пустой **приватный** репозиторий на github.com, затем
`git remote add origin <url> && git push -u origin main`. В репозиторий идёт всё, кроме
секретов — `.gitignore` минимальный (только мусор ОС/инструментов), секретов в проекте нет
вовсе, они хранятся глобально у сотрудника (`~/.config/opp/secrets.env`). → `онбординг.шаг: 6`.

## Шаг 7. Завершение
Отметь в `.статус.yaml`: `онбординг.завершён: true`, `онбординг.шаг: 7`. Сообщи пользователю,
что онбординг завершён, и следующий шаг — положить материалы в «Входные материалы/» (или
приложить в чат) и запустить навык `analyze-materials`.

## Как двигать .статус.yaml
Правь файл через PyYAML, не текстом руками (не сломать структуру):

```bash
"${CLAUDE_PLUGIN_ROOT}/.venv/bin/python" - <<'PY'
import yaml
p = "<проект>/.статус.yaml"
d = yaml.safe_load(open(p, encoding="utf-8")) or {}
d.setdefault("онбординг", {})["шаг"] = <N>
# на шаге 7 дополнительно:
# d["онбординг"]["завершён"] = True
yaml.safe_dump(d, open(p, "w", encoding="utf-8"), allow_unicode=True, sort_keys=False)
PY
```

## Контракт
Вход: вводные (чат + документы), родительская папка проекта. Выход: развёрнутая рабочая папка
с заполненным паспортом, карточкой компании (`SRC-COM`), правилами и оформлением, гайдом
пользователя в корне; `.статус.yaml` с `онбординг.завершён: true`. Возобновляемый: прерванный
онбординг продолжается с шага, записанного в `.статус.yaml`, а не с начала. Git проекта — по
согласию пользователя, репозиторий строго приватный (ЗР-0025). Скилы и инструменты не
копируются в проект — резолвятся из продукта.
